# JVM — Java 虚拟机

> 分类：编程语言
> 标签：JVM, Java 虚拟机, 内存模型, 垃圾回收, GC, 类加载, JVM 调优

## 面试精答

> 适合面试时口述的简洁版本（2-3分钟能说完的要点）

JVM 是 Java 程序的运行环境——它让 Java 实现"一次编译，到处运行"（Write Once, Run Anywhere）。Java 源码编译成字节码（.class 文件），JVM 负责在不同操作系统上解释或编译执行这些字节码。

**JVM 的三大核心模块**：

1. **类加载子系统**：把 .class 文件加载到内存中，经过加载 → 链接（验证、准备、解析）→ 初始化三个阶段
2. **运行时数据区**：JVM 管理的内存区域，包括堆（对象实例）、方法区（类信息、常量）、虚拟机栈（方法调用）、本地方法栈（Native 方法）、程序计数器（当前执行位置）
3. **执行引擎**：负责执行字节码，包含解释器、JIT 编译器（把热点代码编译成机器码提升性能）、垃圾回收器（GC）

**面试最高频的两个话题**：
- **JVM 内存模型**：哪些区域线程共享、哪些私有、什么存在哪里
- **垃圾回收（GC）**：怎么判断对象可回收、有哪些 GC 算法和收集器、怎么调优

## 原理深入详解

> 深入底层原理，理解"为什么"

### 一、前置知识：为什么需要 JVM

**C/C++ 的编译方式**：

```
C 源码 → 编译器 → 机器码（和 CPU 架构绑定）
  - Windows x64 编译的程序 → 只能在 Windows x64 上运行
  - Linux ARM 编译的程序 → 只能在 Linux ARM 上运行
  - 换一个平台就要重新编译
```

**Java 的编译方式**：

```
Java 源码 → javac 编译 → 字节码（.class 文件，和平台无关）
                              ↓
                    Windows JVM → Windows 机器码
                    Linux JVM  → Linux 机器码
                    macOS JVM  → macOS 机器码

字节码是"中间格式"，JVM 是"翻译官"
不同平台装不同的 JVM，同一份 .class 文件到处跑
```

**JVM 本质上是一个软件模拟的计算机**：它有自己的"CPU"（执行引擎）、"内存"（运行时数据区）、"指令集"（字节码）。

### 二、JVM 内存模型（运行时数据区）

这是面试中被问得最多的部分。JVM 把内存分成 5 个区域：

```
┌───────────────────────────────────────────────────┐
│                  JVM 运行时数据区                    │
│                                                   │
│  ┌──────────────────────┐  ┌──────────────────┐  │
│  │   线程共享区域         │  │  线程私有区域      │  │
│  │                      │  │                  │  │
│  │  ┌────────────────┐  │  │ ┌──────────────┐ │  │
│  │  │     堆（Heap）  │  │  │ │虚拟机栈       │ │  │
│  │  │  对象实例、数组  │  │  │ │（VM Stack）   │ │  │
│  │  │  GC 的主要区域   │  │  │ │方法调用栈帧   │ │  │
│  │  └────────────────┘  │  │ └──────────────┘ │  │
│  │                      │  │ ┌──────────────┐ │  │
│  │  ┌────────────────┐  │  │ │本地方法栈     │ │  │
│  │  │  方法区（Method │  │  │ │（Native Stack）│  │
│  │  │   Area/Metaspace│  │  │ └──────────────┘ │  │
│  │  │  类信息、常量池  │  │  │ ┌──────────────┐ │  │
│  │  │  静态变量       │  │  │ │程序计数器     │ │  │
│  │  └────────────────┘  │  │ │（PC Register）│ │  │
│  │                      │  │ └──────────────┘ │  │
│  └──────────────────────┘  └──────────────────┘  │
└───────────────────────────────────────────────────┘
```

#### 1. 堆（Heap）— 面试最核心

**存什么**：几乎所有对象实例和数组都在堆上分配（JIT 优化后的逃逸分析除外）。

**线程共享**：所有线程都能访问堆中的对象。

**GC 的主战场**：垃圾回收主要就是回收堆中不再使用的对象。

**堆的分代结构**（以 JDK 8 默认的 Parallel GC 为例）：

```
堆内存
├── 新生代（Young Generation）— 占堆的 1/3
│   ├── Eden 区（80%）        — 新对象首先分配在这里
│   ├── Survivor 0 区（10%）   — 存活对象的中转站
│   └── Survivor 1 区（10%）   — 存活对象的中转站
│
└── 老年代（Old Generation）— 占堆的 2/3
    — 存活时间长的对象（经过多次 GC 还活着的）

新对象分配流程：
  new Object()
    → Eden 区分配
    → Eden 满了 → Minor GC（清理新生代）
      → 存活的对象复制到 Survivor 区
      → 年龄 +1
      → 年龄达到 15（默认）→ 晋升到老年代
    → 老年代满了 → Full GC（清理整个堆）
```

**为什么分代**：统计学研究发现，大部分对象"朝生夕死"（用完马上就没用了），只有少量对象长期存活。分代后，GC 可以只频繁清理新生代（快），不用每次都扫描整个堆（慢）。

#### 2. 方法区 / 元空间（Metaspace）

**存什么**：类的元数据（类名、方法信息、字段信息）、常量池、静态变量。

**JDK 7 vs JDK 8 的重大变化**：

```
JDK 7 及之前：
  方法区 = 永久代（PermGen），在 JVM 堆内存中
  大小固定（-XX:MaxPermSize），容易 OOM

JDK 8 及之后：
  方法区 = 元空间（Metaspace），使用本地内存（不在 JVM 堆中）
  大小受限于物理内存，不容易 OOM
  字符串常量池移到了堆中

为什么改成元空间：
  永久代大小难以预估（动态生成类多的框架容易撑爆）
  元空间用本地内存，自动扩展，更灵活
```

#### 3. 虚拟机栈（VM Stack）

**存什么**：每个方法调用时创建一个**栈帧（Stack Frame）**，压入栈中。方法返回时弹出。

```
线程 A 的虚拟机栈：

栈顶 →  ┌──────────────────┐
        │ 栈帧3：getOrder()  │ ← 当前执行的方法
        │  局部变量表        │
        │  操作数栈          │
        │  返回地址          │
        ├──────────────────┤
        │ 栈帧2：process()  │
        ├──────────────────┤
        │ 栈帧1：main()     │
        └──────────────────┘

每次方法调用 → 压栈
方法返回    → 弹栈
栈的深度有限 → 递归太深会 StackOverflowError
```

**线程私有**：每个线程有自己的虚拟机栈，互不干扰。

#### 4. 程序计数器（PC Register）

**存什么**：当前线程正在执行的字节码指令的地址（行号）。

**为什么需要它**：CPU 是多线程轮流执行的。线程 A 执行到第 50 行被挂起，切到线程 B，切回来时要记住"上次执行到哪了"——程序计数器就是这个"书签"。

**线程私有**：每个线程有独立的 PC，否则多线程就不知道各自执行到哪了。

#### 5. 本地方法栈

和虚拟机栈类似，但为 Native 方法（Java 调用 C/C++ 代码，如 `System.currentTimeMillis()` 的底层实现）服务。

#### 直接内存（Direct Memory）

不属于 JVM 规范中的运行时数据区，但经常被用到。NIO 的 `ByteBuffer.allocateDirect()` 分配的就是直接内存，不在堆上，不受 GC 管理，但受物理内存限制。

**各区域 OOM 的错误类型**：

| 区域 | 溢出错误 | 常见原因 |
|---|---|---|
| 堆 | `OutOfMemoryError: Java heap space` | 对象太多，内存泄漏 |
| 方法区 | `OutOfMemoryError: Metaspace` | 动态生成太多类（CGLIB 等） |
| 虚拟机栈 | `StackOverflowError` | 递归太深 |
| 直接内存 | `OutOfMemoryError: Direct buffer memory` | NIO 用了太多直接内存 |

### 三、垃圾回收（GC）

#### 怎么判断对象可以回收

**引用计数法（Java 不用）**：每个对象维护一个引用计数器，被引用 +1，引用断开 -1，为 0 就回收。问题：无法解决循环引用（A 引用 B，B 引用 A，计数永远不为 0）。

**可达性分析（Java 用的方案）**：

```
从 GC Roots 出发，沿引用链向下搜索
能到达的对象 = 存活的（标记为可达）
到达不了的   = 可回收的

GC Roots 包括：
  - 虚拟机栈中的局部变量引用
  - 方法区中静态变量引用
  - 方法区中常量引用
  - 本地方法栈中 JNI 引用
  - JVM 内部引用（基本类型对应的 Class 对象等）

示例：

GC Root ──→ Object A ──→ Object B
                              │
                              ↓
                         Object C ←── Object D

A、B、C 都从 Root 可达 → 存活
D 虽然被 C 引用，但 C 不是 Root，等等...
实际上 D 是从 B 可达的（B → C → D），所以 D 也存活

Object E（没有任何引用指向它）→ 不可达 → 可回收
```

#### 四种引用类型

| 引用类型 | 特点 | GC 行为 | 用途 |
|---|---|---|---|
| **强引用** | `Object obj = new Object()` | 只要强引用存在就不会被回收 | 普通变量 |
| **软引用** | `SoftReference<Object>` | 内存不足时才回收 | 缓存 |
| **弱引用** | `WeakReference<Object>` | 下一次 GC 就回收 | ThreadLocal、WeakHashMap |
| **虚引用** | `PhantomReference<Object>` | 随时可能回收，无法获取对象 | 跟踪 GC 回收活动 |

#### GC 算法

**标记-清除（Mark-Sweep）**：

```
标记阶段：遍历所有对象，标记可回收的
清除阶段：清除被标记的对象

优点：实现简单
缺点：产生内存碎片（空闲空间不连续）
      标记和清除效率都不高
```

**标记-复制（Copying）**：

```
把内存分成两块，每次只用一块
GC 时把存活对象复制到另一块，然后清空当前块

Eden + Survivor 就是这个算法：
  Eden + S0 存活对象 → 复制到 S1 → 清空 Eden + S0
  下次 Eden + S1 存活对象 → 复制到 S0 → 清空 Eden + S1

优点：无碎片，分配快（顺序分配）
缺点：浪费一半内存
适合：新生代（对象存活率低，复制少）
```

**标记-整理（Mark-Compact）**：

```
标记阶段：同上
整理阶段：把存活对象向一端移动，然后清理边界以外的空间

优点：无碎片，不浪费空间
缺点：移动对象需要更新引用，耗时
适合：老年代（对象存活率高，不宜用复制算法）
```

#### 常见垃圾回收器

```
                    新生代收集器              老年代收集器
                  ┌─────────────┐       ┌───────────────┐
  JDK 8 默认      │   Parallel   │  ──→ │  Parallel Old  │
                  │   Scavenge   │       │  （吞吐量优先） │
                  └─────────────┘       └───────────────┘

                  ┌─────────────┐       ┌───────────────┐
  追求低延迟       │    ParNew    │  ──→ │     CMS        │
                  │              │       │  （停顿时间短）  │
                  └─────────────┘       └───────────────┘

                  ┌─────────────────────────────────────┐
  JDK 9+ 默认     │              G1                      │
  面向大堆内存     │   （Region 化，兼顾吞吐和延迟）        │
                  └─────────────────────────────────────┘

                  ┌─────────────────────────────────────┐
  JDK 15+         │            ZGC                       │
  超低延迟         │   （亚毫秒级停顿，TB 级堆）           │
                  └─────────────────────────────────────┘
```

**G1 的核心设计**：

```
传统收集器：堆是连续的新生代 + 老年代
G1：把堆分成多个等大的 Region（默认 2048 个）

每个 Region 可以是 Eden / Survivor / Old / Humongous（大对象）
G1 维护每个 Region 的回收价值（垃圾越多价值越高）
优先回收价值高的 Region → "Garbage First" 的由来

优势：
  - 可以设定期望的停顿时间（-XX:MaxGCPauseMillis=200）
  - 大堆（6GB+）表现比 CMS 好
  - 不存在内存碎片
```

### 四、类加载机制

#### 类加载的五个阶段

```
                    ┌──────────┐    ┌──────────┐    ┌──────────┐
  .class 文件 ────→ │  加载     │───→│  链接     │───→│  初始化   │
                    │ Loading  │    │ Linking  │    │Initialize│
                    └──────────┘    └──────────┘    └──────────┘
                        │         ┌────┼────┐
                        │         ↓    ↓    ↓
                        │      验证  准备  解析
                        │    Verify Prepare Resolve
                        ↓
                   读取 .class
                   文件到内存
                   生成 Class
                   对象
```

**加载**：通过类的全限定名找到 .class 文件，读取二进制数据到内存。

**验证**：检查字节码是否符合 JVM 规范（防止恶意构造的 .class 文件导致 JVM 崩溃）。

**准备**：为静态变量分配内存并赋零值（`int → 0`，`boolean → false`，`引用 → null`）。注意不是代码中写的初始值。

**解析**：把常量池中的符号引用（类名字符串）替换为直接引用（内存地址指针）。

**初始化**：执行类的 `<clinit>()` 方法——即静态变量赋值和 static 代码块的执行。

#### 双亲委派模型

```
类加载请求自底向上委派，加载动作自顶向下尝试：

        ┌────────────────────┐
        │  Bootstrap ClassLoader │ ← 加载 rt.jar（Java 核心类库）
        │  （启动类加载器）        │   C++ 实现，不是 Java 类
        └──────────┬─────────┘
                   │ 父加载器
        ┌──────────┴─────────┐
        │ Extension ClassLoader │ ← 加载 ext 目录的扩展类
        │ （扩展类加载器）        │
        └──────────┬─────────┘
                   │ 父加载器
        ┌──────────┴─────────┐
        │  Application ClassLoader │ ← 加载 classpath 上的类（你的代码）
        │  （应用类加载器）           │
        └──────────┬─────────┘
                   │ 父加载器
        ┌──────────┴─────────┐
        │   Custom ClassLoader   │ ← 你自己写的类加载器
        │   （自定义类加载器）      │
        └────────────────────┘

加载 java.lang.String 时：
  Custom → 问 App → 问 Ext → 问 Bootstrap → Bootstrap 能加载 → 由 Bootstrap 加载

为什么叫"双亲委派"：
  子加载器不自己加载，先委派给父加载器
  父加载器搞不定才自己来

为什么需要双亲委派：
  1. 安全：防止用户自己写一个 java.lang.String 替换核心类
  2. 避免重复加载：同一个类只被加载一次

什么场景会打破双亲委派：
  - SPI 机制（JDBC）：核心类需要加载用户提供的实现类
  - Tomcat：不同 Web 应用可以加载同一个类的不同版本
  - OSGi：模块化框架需要自己的类加载策略
```

### 五、JIT 编译器

**为什么 Java 不直接解释执行字节码**：解释执行是逐条翻译字节码为机器码再执行，很慢。

**JIT（Just-In-Time）编译器的作用**：把频繁执行的"热点代码"直接编译成本地机器码，后续执行时不再解释，直接运行机器码，性能接近 C/C++。

```
解释执行（慢）：
  字节码 → 解释器逐条翻译 → 机器码 → 执行
  每次执行都要翻译

JIT 编译（快）：
  字节码 → JIT 编译器 → 机器码（缓存起来）
  后续执行直接用缓存的机器码

热点检测：
  JVM 对每个方法的调用次数计数
  调用次数超过阈值（默认 10000 次）→ 触发 JIT 编译
  这就是为什么 Java 程序"越跑越快"——热点代码被 JIT 优化了
```

### 六、常见 JVM 调优参数

| 参数 | 含义 | 典型值 |
|---|---|---|
| `-Xms` | 堆初始大小 | 和 `-Xmx` 设一样（避免运行时动态扩缩） |
| `-Xmx` | 堆最大大小 | 物理内存的 50-70% |
| `-Xmn` | 新生代大小 | 堆的 1/3 到 1/2 |
| `-Xss` | 每个线程栈大小 | 256K~1M |
| `-XX:MetaspaceSize` | 元空间初始大小 | 256M |
| `-XX:MaxMetaspaceSize` | 元空间最大大小 | 512M |
| `-XX:NewRatio` | 老年代:新生代比例 | 2（老年代是新生代的 2 倍） |
| `-XX:SurvivorRatio` | Eden:Survivor 比例 | 8（Eden 是 Survivor 的 8 倍） |
| `-XX:+UseG1GC` | 使用 G1 垃圾回收器 | JDK 9+ 默认 |
| `-XX:MaxGCPauseMillis` | G1 目标最大停顿时间 | 200ms |
| `-XX:+PrintGCDetails` | 打印 GC 详细日志 | 调优时必开 |

## 常见追问与应对

**Q1: 什么时候对象会进入老年代？**
> 四种情况：1）年龄达到阈值（默认 15，经过 15 次 Minor GC 还存活）；2）大对象直接分配到老年代（`-XX:PretenureSizeThreshold`，避免新生代复制开销）；3）Survivor 区放不下的对象直接进入老年代；4）动态年龄判断——Survivor 中相同年龄的对象总大小超过 Survivor 空间的一半，该年龄及以上的对象直接进入老年代。

**Q2: Full GC 太频繁怎么排查？**
> 排查步骤：1）看 GC 日志，确认 Full GC 的频率和耗时；2）`jstat -gcutil` 看各区域使用率；3）`jmap -histo` 看哪些对象占内存最多；4）`jmap -dump` 导出堆快照，用 MAT（Memory Analyzer Tool）分析哪个类的对象最多、谁持有了它们的引用；5）常见原因：内存泄漏（集合只加不删）、大对象分配、Metaspace 溢出（动态生成太多类）。

**Q3: JVM 为什么要有 JIT？为什么不一开始就全部编译成机器码？**
> 全部编译有两个问题：1）启动慢——Java 程序可能几千个类，全编译完再运行启动要几十秒；2）浪费——大部分代码只执行一两次（如初始化代码），编译它们白白浪费时间。JIT 只编译"热点代码"是性能和启动时间的折中。

**Q4: Java 9+ 的模块化对类加载有什么影响？**
> Java 9 引入了模块系统（JPMS），每个模块有明确的依赖声明和导出控制。模块路径（module path）替代了部分 classpath 的作用，类的可见性从"类路径上都能看到"变成了"模块显式导出才能看到"。类加载器也做了调整，但双亲委派的基本模型没变。
