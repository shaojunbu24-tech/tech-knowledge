# Java GC 垃圾回收详解

> 分类：编程语言
> 标签：Java, GC, 垃圾回收, 垃圾收集器, G1, ZGC, CMS, JVM 调优, 内存管理

## 面试精答

> 适合面试时口述的简洁版本（2-3分钟能说完的要点）

Java GC 的核心问题是三个：**什么对象该回收、用什么算法回收、用什么收集器来执行**。

**判断对象是否可回收**：用"可达性分析"——从 GC Roots 出发沿引用链搜索，不可达的对象就是可回收的。GC Roots 包括：栈帧中的局部变量引用、静态变量引用、JNI 引用、锁持有对象等。

**四种 GC 算法**：标记-清除（碎片多）、标记-整理（慢但无碎片）、复制算法（快但浪费空间）、分代收集（新生代用复制，老年代用标记-整理）。分代收集的核心依据是"弱代假说"——绝大多数对象朝生夕死，少数长期存活。

**六种主流收集器**：
- **Serial**：单线程，适合客户端
- **Parallel**：吞吐量优先，JDK 8 默认
- **CMS**：低停顿，已废弃（标记-清除有碎片）
- **G1**：_region 化分代，停顿可控，JDK 9+ 默认_
- **ZGC**：亚毫秒停顿，JDK 15+ 生产可用
- **Shenandoah**：和 ZGC 类似的超低延迟方案

**JDK 版本差异**：JDK 8 默认 Parallel GC → JDK 9 默认 G1 → JDK 11 G1 成熟 + ZGC 实验性引入 → JDK 15 ZGC 生产可用 → JDK 17 LTS 推荐 G1 或 ZGC → JDK 21 ZGC 分代模式 + 虚拟线程配合。

## 原理深入详解

> 深入底层原理，理解"为什么"

### 一、为什么需要垃圾回收

在 C/C++ 中，程序员手动管理内存——用 `malloc` 分配，用 `free` 释放。忘了释放就**内存泄漏**，释放了还在用就**悬垂指针**（use-after-free）。这是 C/C++ 最常见、最危险的 bug 来源之一。

Java 的方案是**让 JVM 自动管理内存**：程序员只管 `new` 对象，不用手动释放，JVM 的垃圾回收器（Garbage Collector, GC）会自动找出不再使用的对象并回收内存。

**代价**：GC 不是免费的——它需要 CPU 时间来执行回收，且回收期间可能暂停应用线程（即 **STW, Stop-The-World**）。所以 GC 研究的核心目标就是：**在尽可能短的停顿时间内，回收尽可能多的垃圾**。

### 二、JVM 堆内存的分代结构

GC 的主战场是**堆（Heap）**。JVM 把堆分成不同的"代"，基于一个重要的统计规律：

**弱代假说（Weak Generational Hypothesis）**：
- **弱代假说 1**：绝大多数对象"朝生夕死"——创建后很快就没用了（比如方法内的局部变量、临时字符串）
- **弱代假说 2**：熬过多次 GC 的对象往往会继续存活很久（比如缓存、连接池、Spring Bean）

根据这个规律，JVM 把堆分成**新生代（Young Generation）** 和**老年代（Old Generation）**：

```
┌─────────────────── 堆（Heap）──────────────────────┐
│                                                     │
│  新生代（Young Gen）              老年代（Old Gen）  │
│  ┌──────────┬──────────┐        ┌──────────────┐   │
│  │  Eden    │ Survivor │        │              │   │
│  │          │  S0  | S1│        │  Tenured     │   │
│  │          │          │        │              │   │
│  └──────────┴──────────┘        └──────────────┘   │
│   默认比例 8 : 1 : 1              默认占堆的 2/3     │
│   新对象先分配在 Eden              存活久的对象晋升   │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**对象的生命旅程**：

```
新对象 → 分配在 Eden 区
          │
          Eden 满 → Minor GC（只清理新生代）
          │
          ├─ 不存活 → 回收
          │
          └─ 存活 → 复制到 Survivor 区（S0 或 S1）
                     │
                     每次在 S0 和 S1 之间来回复制
                     每熬过一次 Minor GC，年龄 +1
                     │
                     年龄达到阈值（默认 15）→ 晋升到老年代
                     │
                     老年代满 → Full GC（清理整个堆 + 方法区）
```

**为什么要用两个 Survivor 区**：复制算法需要一块空的区域来放存活对象。两个 Survivor 区交替使用——一个放存活对象，另一个在这次 GC 中保持为空，下次交换角色。这样避免了内存碎片。

**为什么 Eden:Survivor:S0:S1 = 8:1:1**：根据统计，新生代中 98% 的对象在一次 Minor GC 中就会被回收。所以 80% 的空间给 Eden（大部分对象在这里创建然后立刻死去），只有 10% 的空间就够放存活的对象。

### 三、判断对象是否可回收

#### 3.1 引用计数法（Java 不用这种方法）

最直观的想法：给每个对象维护一个"被引用次数"。被引用 +1，引用断开 -1，计数为 0 就回收。

```
问题：循环引用

Object A ──ref──→ Object B
    ↑                  │
    └──── ref ─────────┘

A 引用 B，B 引用 A
两者的引用计数都不为 0，但实际上已经没有外部引用了
→ 内存泄漏，永远无法回收
```

所以 Java **不使用**引用计数法。

#### 3.2 可达性分析（Java 实际使用的方法）

从一组确定的起点（**GC Roots**）出发，沿着引用关系向下搜索。如果一个对象到所有 GC Roots 都没有引用链，说明这个对象不可达，可以回收。

```
GC Roots ──→ A ──→ B ──→ C
               │
               └──→ D

                 E ──→ F

搜索结果：
  A, B, C, D 可达 → 不回收
  E, F 不可达（没有 GC Root 能走到它们）→ 可以回收
```

**什么是 GC Roots**：

```
GC Roots 包含：

1. 虚拟机栈中的引用
   — 当前正在执行的方法中的局部变量引用的对象
   — 比如方法里写了 User u = new User()，u 就是栈上的引用

2. 方法区中的静态变量引用
   — 类的 static 字段引用的对象
   — 比如 private static Cache cache = new Cache()

3. 方法区中的常量引用
   — 字符串常量池中的引用

4. 本地方法栈中的 JNI 引用
   — Native 方法（C/C++ 写的代码）引用的 Java 对象

5. 同步锁（synchronized）持有的对象

6. JVM 内部引用
   — 基本数据类型的 Class 对象
   — 常驻异常（NullPointerException 等）
   — 类加载器
```

#### 3.3 四种引用类型

Java 提供了四种引用强度不同的类型，让开发者可以精细控制对象的生命周期：

| 引用类型 | 类 | GC 行为 | 典型用途 |
|---|---|---|---|
| **强引用** | `Object obj = new Object()` | 只要有引用就永远不会回收 | 日常编程 |
| **软引用** | `SoftReference<Object>` | 内存不足时才回收 | 缓存（内存紧张时自动清理） |
| **弱引用** | `WeakReference<Object>` | 下一次 GC 就回收 | ThreadLocal、WeakHashMap |
| **虚引用** | `PhantomReference<Object>` | 随时可能回收，无法获取对象 | 跟踪 GC 回收活动、管理堆外内存 |

```
为什么 ThreadLocal 用弱引用？

ThreadLocalMap 的 key 是 ThreadLocal 对象的弱引用。
当 ThreadLocal 变量设为 null 时，key 被 GC 回收，
value 在下次 get/set 时被清理。
如果 key 是强引用，ThreadLocal 对象就永远无法被回收 → 内存泄漏。

注意：即使如此，ThreadLocal 使用后仍应调用 remove()，
因为 value 依然是强引用，不手动 remove 会泄漏。
```

### 四、四种 GC 算法

#### 4.1 标记-清除（Mark-Sweep）

```
阶段一：标记
  从 GC Roots 遍历，标记所有可达对象

阶段二：清除
  遍历整个堆，回收未标记的对象

执行前：
  [A][B][空][C][空][D][空][E]
   ✓  ✓       ✓       ✓

执行后：
  [A][B][空][空][空][D][空][空]

优点：实现简单
缺点：
  1. 内存碎片 — 清除后留下不连续的空闲块
     后续分配大对象时找不到连续空间，虽然总空闲够
  2. 效率不高 — 需要遍历整个堆两次（标记一次、清除一次）
```

#### 4.2 标记-整理（Mark-Compact）

```
阶段一：标记（和标记-清除一样）

阶段二：整理
  把所有存活对象移动到内存的一端，然后清理边界以外的空间

执行前：
  [A][B][空][C][空][D][空][E]
   ✓  ✓       ✓       ✓

执行后：
  [A][B][C][D][E][空][空][空]
  ← 存活对象紧凑排列 → ← 大块连续空闲 →

优点：没有内存碎片
缺点：移动对象需要更新所有引用指向新地址，耗时较长
```

#### 4.3 复制算法（Copying）

```
把内存分成两块，每次只用其中一块：

From 区（当前使用）              To 区（空闲）
[A][B][C][D][空][空]          [空][空][空][空][空][空]
 ✓     ✓

GC 时：把存活对象复制到 To 区，然后清空 From 区

From 区                        To 区
[空][空][空][空][空][空]       [A][C][空][空][空][空]
                                ← 存活对象紧凑排列

然后交换 From 和 To 的角色

优点：
  1. 没有碎片 — 存活对象紧凑排列
  2. 速度快 — 只需要复制少量存活对象（新生代 98% 都会死）
  3. 分配快 — 内存连续，指针碰撞即可分配

缺点：
  浪费一半内存 — 每次只能用总空间的一半
```

新生代为什么适合复制算法：因为 98% 的对象会在一次 Minor GC 中被回收，只需要复制 2% 的存活对象，代价极低。Eden + 一个 Survivor = 90% 的空间可用，浪费的只有 10%（另一个 Survivor）。

#### 4.4 分代收集算法

不是一种新算法，而是**根据不同代的特点选择不同算法**的策略：

```
新生代：
  特点：对象朝生夕死，每次 GC 回收 90%+
  算法：复制算法
  原因：存活少，复制代价低；没有碎片，分配快

老年代：
  特点：对象存活率高，GC 频率低
  算法：标记-整理 或 标记-清除
  原因：存活对象多，复制代价太高；用标记-整理避免碎片
```

### 五、六种主流垃圾收集器

收集器是 GC 算法的具体实现。不同的收集器在**吞吐量**和**停顿时间**之间做不同的权衡。

```
收集器演进路线：

JDK 1.3  Serial                          ← 单线程，最基础
           │
JDK 1.4  Parallel Scavenge / Serial Old   ← 多线程，吞吐量优先
           │
JDK 1.5  CMS                              ← 低停顿，并发标记
           │
JDK 1.7  G1                                ← Region 化，停顿可控
           │
JDK 11   G1 (默认) + ZGC (实验性)          ← ZGC 亚毫秒停顿
           │
JDK 15   ZGC (生产可用)                    ← 最大支持 TB 级堆
           │
JDK 17   G1 (默认 LTS) / ZGC (推荐)       ← LTS 版本
           │
JDK 21   ZGC 分代模式                      ← 进一步优化
```

#### 5.1 Serial 收集器

```
特点：单线程执行 GC

  应用线程：███░░░░░░░░░███
                     ↑
              STW：单个 GC 线程工作
              应用全部暂停

优点：简单高效（没有多线程交互开销）
缺点：STW 停顿时间长
适用：客户端模式、小堆（< 100MB）
参数：-XX:+UseSerialGC
```

#### 5.2 Parallel Scavenge / Parallel Old

```
特点：多线程执行 GC，目标是最大化吞吐量
  吞吐量 = 应用运行时间 / (应用运行时间 + GC 时间)

  应用线程：██████░░░░██████
                     ↑
              STW：多个 GC 线程并行工作
              比 Serial 快，但仍然会暂停应用

JDK 8 的默认收集器组合：
  新生代：Parallel Scavenge（复制算法）
  老年代：Parallel Old（标记-整理）

参数：
  -XX:+UseParallelGC
  -XX:MaxGCPauseMillis=200    ← 目标最大停顿时间
  -XX:GCTimeRatio=99          ← 目标吞吐量（99 = 99/100 时间在跑应用）

适用：批处理、离线计算等对吞吐量敏感的场景
```

#### 5.3 CMS（Concurrent Mark Sweep）

```
目标：最小化 STW 停顿时间，让 GC 和应用并发执行

四个阶段：

1. 初始标记（STW，极短）
   只标记 GC Roots 直接引用的对象
   ─── 耗时极短，因为只遍历根引用

2. 并发标记（与应用并发执行）
   从 GC Roots 遍历整个引用链
   ─── 耗时最长，但应用还在跑

3. 重新标记（STW，较短）
   修正并发标记期间因应用运行导致的变化
   ─── 用"增量更新"算法处理变化

4. 并发清除（与应用并发执行）
   清除未标记的对象
   ─── 因为用的是标记-清除，不需要移动对象

时间线：
  应用：  ████░░░████████░░████████
  GC：       ↑↑   ↑↑↑↑↑↑↑↑   ↑↑
          初始  并发标记    重新  并发清除
          标记            标记

优点：停顿时间短（两次 STW 都很短）
缺点：
  1. 内存碎片 — 用标记-清除，不整理
  2. CPU 敏感 — 并发阶段占用 CPU，默认启动 (CPU数+3)/4 个 GC 线程
  3. 浮动垃圾 — 并发标记阶段应用新产生的垃圾，只能下次 GC 再清
  4. Concurrent Mode Failure — 老年代预留空间不够给浮动垃圾，
     触发 Full GC（退回 Serial Old 收集，停顿很长）

命运：JDK 9 标记废弃，JDK 14 正式移除
参数：-XX:+UseConcMarkSweepGC（已不可用）
```

#### 5.4 G1（Garbage First）

G1 是 JDK 9 及之后的默认收集器，是 JVM GC 的一个重大设计转变。

**核心设计变化 — Region 化**：

```
传统收集器：新生代和老年代是物理连续的整块内存

  [========== 新生代 ==========][============== 老年代 ==============]

G1：把堆划分成大小相等的 Region（默认 2048 个）

  ┌────┬────┬────┬────┬────┬────┬────┬────┐
  │ E  │ S  │ O  │ E  │ H  │ O  │ E  │ E  │
  ├────┼────┼────┼────┼────┼────┼────┼────┤
  │ O  │ E  │ E  │ S  │ O  │ E  │ H  │ O  │
  ├────┼────┼────┼────┼────┼────┼────┼────┤
  │ E  │ O  │ E  │ E  │ E  │ O  │ E  │ S  │
  └────┴────┴────┴────┴────┴────┴────┴────┘

  E = Eden Region    S = Survivor Region
  O = Old Region     H = Humongous Region（大对象）

特点：
  1. 新生代和老年代不再是连续的，而是 Region 的逻辑集合
  2. 每个 Region 可以动态地在 E/S/O 之间转换
  3. 大对象（超过 Region 大小一半）直接分配在 Humongous Region
  4. 回收以 Region 为单位，优先回收垃圾最多的 Region（Garbage First 名字由来）
```

**G1 的回收流程**：

```
Young GC（只回收新生代 Region）：
  多线程并行，会 STW，但因为有 RSet 不需要扫描整个堆

Mixed GC（回收所有新生代 + 垃圾最多的老年代 Region）：
  这是 G1 的核心优势——不回收整个老年代，只回收"最值得回收的"部分

  步骤：
  1. 初始标记（STW，极短，借助 Young GC 完成）
  2. 并发标记（与应用并发，找出整个堆的存活对象）
  3. 最终标记（STW，处理并发阶段的变化）
  4. 筛选回收（STW，按回收价值排序 Region，选最值得的回收）

关键数据结构 — Remembered Set（RSet）：
  每个 Region 维护一个 RSet，记录"其他哪些 Region 引用了我"
  → GC 时只需要关注 RSet 中的引用，不用扫描整个堆
  → 代价：RSet 本身占额外内存（堆的 10%-20%）
```

**G1 的可预测停顿**：

```
-XX:MaxGCPauseMillis=200（默认 200ms）

G1 根据历史数据预测每个 Region 的回收耗时
在允许的停顿时间内，尽量多回收垃圾最多的 Region

比如 200ms 内能回收 5 个 Region
那就选垃圾占比最高的 5 个回收
如果垃圾不够多，这次就不回收这些 Region

这就是"Garbage First"的含义：优先回收垃圾最多的 Region
```

```
G1 调优关键参数：
  -XX:+UseG1GC                    启用 G1
  -XX:MaxGCPauseMillis=200        目标最大停顿时间
  -XX:G1HeapRegionSize            Region 大小（1/2/4/8/16/32MB）
  -XX:InitiatingHeapOccupancyPercent=45  触发并发标记的堆占用阈值
  -XX:G1MixedGCCountTarget=8      Mixed GC 后最多多少次 Mixed GC
```

#### 5.5 ZGC（Z Garbage Collector）

ZGC 是 Oracle 开发的超低延迟收集器，目标是将 GC 停顿控制在**亚毫秒级别**（< 1ms），且停顿时间不随堆大小增长。

**核心设计 — 着色指针 + 读屏障**：

```
传统 GC 用写屏障（Write Barrier）来跟踪引用变化
ZGC 用读屏障（Load Barrier）+ 着色指针（Colored Pointers）

着色指针：
  在 64 位对象的指针中，借用几个 bit 来存储"颜色"信息
  表示这个引用指向的对象当前处于什么状态

  ┌──────────────────────────────────────────────────────────┐
  │  62  61  60  59  58  │        43 ... 0                   │
  │  Finalized Remap  │  Marked1  Marked0  │   对象地址       │
  │    (颜色位)        │                   │                  │
  └──────────────────────────────────────────────────────────┘

读屏障：
  每次应用线程从堆中加载引用时，ZGC 检查指针颜色
  如果颜色不对 → 说明对象正在被移动 → 先修正指针再返回
  → 对象移动和应用的运行完全并发
  → 不需要 STW 来移动对象
```

**ZGC 的回收流程**：

```
三个阶段几乎全部并发执行：

1. 标记（并发）
   从 GC Roots 开始遍历，标记存活对象
   通过着色指针的 Marked0/Marked1 位来判断对象是否已标记

2. 转移（并发）
   把存活对象从垃圾多的 Region 转移到新 Region
   应用线程通过读屏障自纠正指针

3. 重定位（并发）
   更新所有指向已转移对象的引用

时间线：
  应用：████████████████████████████
  GC：   标记标记标记  转移转移转移  重定位重定位

  几乎完全并发，STW 只有：
  - 初始标记根引用（亚毫秒）
  - 再标记少量根引用（亚毫秒）

性能：
  停顿时间 < 1ms（通常 0.1-0.5ms）
  不受堆大小影响（10GB 堆和 10TB 堆的停顿一样短）
  支持 TB 级堆
```

**ZGC 的版本演进**：

```
JDK 11：ZGC 实验性引入（非分代模式）
  只有一套 Region，不做新生代/老年代区分
  → 短命对象和长寿对象混在一起，每次都扫描全部

JDK 13：最大堆支持从 4TB 提升到 16TB

JDK 15：ZGC 正式成为生产可用特性（Production Ready）
  -XX:+UseZGC

JDK 16：并发线程栈扫描（进一步减少 STW）

JDK 21：ZGC 分代模式（Generational ZGC）
  引入新生代/老年代区分
  短命对象在新生代快速回收，不用每次都扫全部
  大幅提升吞吐量
  -XX:+UseZGC -XX:+ZGenerational
```

#### 5.6 Shenandoah

Shenandoah 是 Red Hat 开发的超低延迟收集器，目标与 ZGC 类似。

```
核心区别 vs ZGC：
  ZGC：着色指针 + 读屏障
  Shenandoah：Brooks Pointer（转发指针）+ 读/写屏障

Brooks Pointer：
  每个对象头部多一个转发指针，指向对象当前的实际位置
  对象被移动时，旧位置留下转发指针
  访问旧地址时，通过转发指针找到新地址

性能：
  停顿时间通常 < 10ms，与堆大小无关
  比 ZGC 稍慢（因为写屏障开销），但实现更简单

版本：
  JDK 12 实验性引入
  JDK 15 生产可用
  参数：-XX:+UseShenandoahGC
```

### 六、JDK 版本 GC 差异对比

这是面试中经常被追问的重点——不同 JDK 版本之间 GC 的默认行为和推荐策略有什么变化。

| 维度 | JDK 8 | JDK 11 | JDK 17 (LTS) | JDK 21 (LTS) |
|---|---|---|---|---|
| **默认 GC** | Parallel GC | G1 | G1 | G1 |
| **推荐 GC** | G1（手动开启） | G1 | G1 / ZGC | G1 / ZGC 分代 |
| **CMS** | 可用 | 废弃 | 移除 | 移除 |
| **ZGC** | 不支持 | 实验性 | 实验性 | 分代模式生产可用 |
| **Shenandoah** | 不支持 | 实验性 | 可用 | 可用 |
| **关键特性** | Lambda、Stream | HTTP Client、var | sealed class、Pattern Matching | 虚拟线程、Record Pattern |
| **堆默认大小** | 物理内存 1/4（最小 8MB） | 同左 | 同左 | 同左 |
| **GC 日志参数** | `-XX:+PrintGCDetails` | `-Xlog:gc*` | `-Xlog:gc*` | `-Xlog:gc*` |

**迁移建议**：

```
JDK 8 → JDK 11（最常见的升级路径）：
  1. GC 从 Parallel 切换到 G1
     → 大多数场景不需要额外调参
     → 关注 -XX:MaxGCPauseMillis 是否满足需求
  2. GC 日志参数变化：
     -XX:+PrintGCDetails -XX:+PrintGCTimeStamps
     → 变为 -Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=10M
  3. 注意 CMS 已废弃，如果之前用了 CMS 需要迁移到 G1

JDK 11 → JDK 17（LTS 到 LTS 升级）：
  1. G1 已成熟，继续使用
  2. 如果延迟要求极高（< 10ms），考虑 ZGC
  3. 注意一些内部 API 被强封装（--add-opens）

JDK 17 → JDK 21：
  1. ZGC 分代模式是最大亮点
  2. 虚拟线程对 GC 的影响：
     虚拟线程的栈帧存在堆上（而不是 native 线程的栈上）
     大量虚拟线程 = 大量堆内存分配
     → 需要更大的堆或更频繁的 GC
     → ZGC 的亚毫秒停顿非常适合虚拟线程场景
```

### 七、对象从新生代到老年代的四种晋升路径

```
路径 1：年龄达到阈值
  对象在 Survivor 区每熬过一次 Minor GC，年龄 +1
  达到 -XX:MaxTenuringThreshold（默认 15）→ 晋升到老年代
  实际上不一定等到 15 岁：

路径 2：动态年龄判断
  Survivor 区中，某个年龄的所有对象总大小超过 Survivor 空间的一半
  → 该年龄及以上的所有对象直接晋升到老年代
  这是 JVM 的自适应策略，避免 Survivor 区溢出

路径 3：Survivor 空间不足
  Minor GC 后存活对象太多，Survivor 放不下
  → 放不下的部分直接进入老年代（担保机制）

路径 4：大对象直接进老年代
  超过 -XX:PretenureSizeThreshold 的对象直接在老年代分配
  避免在 Eden 和 Survivor 之间来回复制的大开销
  例如：大数组、大字符串、大 ByteBuffer

为什么要尽量减少大对象：
  1. 大对象直接进老年代，快速填满老年代
  2. 触发 Full GC，停顿时间长
  3. 如果大对象又是短命的，浪费老年代空间
```

### 八、Full GC 的触发条件与排查

```
Full GC 是最重的 GC 操作，会 STW 并清理整个堆 + 方法区

触发条件：
  1. 老年代空间不足
     — 大量对象晋升 / 大对象直接分配
  2. Metaspace 空间不足
     — 动态生成了太多类（CGLIB、反射代理）
  3. System.gc() 被调用
     — 建议性调用，不一定会触发，但通常会
  4. CMS 的 Concurrent Mode Failure
     — CMS 并发阶段老年代空间不够放浮动垃圾
  5. G1 的 Evacuation Failure
     — Mixed GC 时没有足够的空闲 Region 放存活对象

排查 Full GC 的步骤：

1. 看 GC 日志（确认频率和耗时）
   jcmd <pid> GC.log_filename 或查看启动参数中的 gc.log 文件

2. jstat 实时监控各区域使用率
   jstat -gcutil <pid> 1000
   关注 FGC（Full GC 次数）和 FGCT（Full GC 总耗时）

3. jmap 看对象分布
   jmap -histo <pid> | head -20
   找出哪些类的对象占内存最多

4. 导出堆快照做离线分析
   jmap -dump:format=b,file=heap.hprof <pid>
   用 Eclipse MAT 或 VisualVM 分析

5. 常见原因
   - 内存泄漏：集合只加不删、ThreadLocal 没 remove
   - 大对象：一次性加载大量数据到内存
   - Metaspace：动态代理/CGLIB 生成了太多类
```

### 九、GC 调优实战

#### 调优目标

```
先定目标，再调参。常见三种目标：

1. 吞吐量优先（批处理、离线分析）
   目标：GC 时间占总时间比例最小
   推荐：Parallel GC
   参数：-XX:GCTimeRatio=99 → 目标 GC 不超过 1% 时间

2. 延迟优先（Web 服务、API 网关、交易系统）
   目标：单次 GC 停顿最短
   推荐：G1（停顿 < 200ms）或 ZGC（停顿 < 1ms）
   参数：-XX:MaxGCPauseMillis=50

3. 资源均衡（通用场景）
   推荐：G1（JDK 9+ 默认就是它，大多数场景够用）
```

#### G1 调优实践

```
原则：G1 已经很智能了，大多数情况只需要设三个参数

java -Xmx4g -Xms4g \
     -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=100 \
     -Xlog:gc*:file=gc.log \
     -jar app.jar

进阶调优（只在有明确问题时才动）：

1. Mixed GC 触发太晚 → 老年代堆积
   -XX:InitiatingHeapOccupancyPercent=35（默认 45，提前触发）

2. Mixed GC 回收太少 → 老年代回收不充分
   -XX:G1MixedGCLiveThresholdPercent=80（默认 85，Region 存活率低于此值才回收）

3. Humongous 对象太多 → 直接进老年代
   -XX:G1HeapRegionSize=16m（增大 Region，减少大对象）

4. 晋升太快 → 年轻对象过早进老年代
   -XX:MaxTenuringThreshold=20（默认 15，让对象在新生代多待会）
```

## 常见追问与应对

**Q1: G1 和 CMS 的核心区别是什么？为什么 CMS 被淘汰了？**
> 三个核心区别：1）**内存布局不同**——CMS 的分代是物理连续的整块，G1 把堆分成 Region，分代是逻辑概念；2）**回收策略不同**——CMS 每次都回收整个老年代，G1 只回收垃圾最多的 Region（Mixed GC）；3）**碎片处理不同**——CMS 用标记-清除不整理碎片，长期运行后碎片化严重触发 Full GC，G1 用复制算法移动存活对象，天然无碎片。CMS 被淘汰的原因就是碎片问题和 Concurrent Mode Failure 导致的长时间 STW，G1 从根本上解决了这两个问题。

**Q2: ZGC 怎么做到停顿时间不受堆大小影响的？**
> 核心是**着色指针 + 读屏障**的组合。传统 GC 在移动对象时需要 STW，是因为移动后要更新所有指向旧地址的引用——这需要暂停应用，否则应用可能访问到旧地址。ZGC 的方案是：移动对象时在旧地址留下"颜色标记"，应用线程通过读屏障在每次加载引用时检查颜色，如果发现对象正在被移动，就自纠正指针指向新地址。这样移动操作就完全不需要暂停应用了。代价是每次加载引用多了一次检查（读屏障的开销），但这个开销极小（通常 < 4% 吞吐量损失）。

**Q3: 什么场景该选 ZGC 而不是 G1？**
> 三个判断条件，满足任一就该考虑 ZGC：1）**延迟 SLA 非常严格**——要求 P99 延迟 < 50ms，G1 的 100-200ms 停顿可能不够；2）**堆非常大**——超过 32GB，G1 的 Mixed GC 停顿会随堆增大而增长，ZGC 不会；3）**实时交易系统**——金融交易、广告竞价等，任何一次长时间停顿都可能导致交易超时或报价失败。反过来，如果堆 < 8GB 或对吞吐量要求极高（批处理），G1 更合适。

**Q4: Minor GC 和 Full GC 的区别？**
> Minor GC 只清理新生代（Eden + Survivor），触发条件是 Eden 区满。特点：频率高（可能几秒一次）、速度快（通常 10-50ms）、STW 时间短。Full GC 清理整个堆（新生代 + 老年代）+ 方法区/Metaspace，触发条件是老年代满或 Metaspace 满。特点：频率低、速度慢（可能几百毫秒到几秒）、STW 时间长。调优的核心目标之一就是**减少 Full GC 的频率**。

**Q5: JDK 8 升级到 JDK 17，GC 方面要注意什么？**
> 四个关键点：1）**默认 GC 变了**——从 Parallel 切换到 G1，大部分场景性能会变好，但吞吐量敏感的批处理任务可能需要验证；2）**GC 日志参数变了**——`-XX:+PrintGCDetails` 要改成 `-Xlog:gc*`；3）**CMS 不能用了**——JDK 14 已经移除，之前用 CMS 的必须迁移到 G1；4）**新增 Metaspace 回收能力**——JDK 8 的 Metaspace 几乎不回收（类卸载困难），JDK 17 改善了类卸载，Metaspace 溢出问题减少。

**Q6: 为什么 Survivor 区只有 10% 的空间就够用？**
> 因为弱代假说的统计规律——新生代中 98% 的对象会在一次 Minor GC 中被回收。假设 Eden 有 80MB，Minor GC 时平均只有 2% 即 1.6MB 的对象存活，10MB 的 Survivor 完全够放。如果某次 GC 存活对象突然很多（超过 Survivor 容量），放不下的部分会通过"担保机制"直接进入老年代。

## 完整面试回答

> 模拟面试中的实际对话，展示如何从头到尾回答 GC 相关问题

**面试官**：能聊聊 Java 的垃圾回收机制吗？

**候选人**：好的。Java GC 的核心思路是自动管理内存——程序员不需要手动释放对象，JVM 会自动找出不再使用的对象并回收。

判断一个对象能不能回收，JVM 用的是**可达性分析**。从一组确定的起点叫 GC Roots 出发，沿着引用关系往下搜索。如果一个对象到所有 GC Roots 都没有引用链可达，说明没有人在用它了，就可以回收。GC Roots 包括栈帧中的局部变量引用、静态变量引用、JNI 引用等。这里顺便提一下，Java 不用引用计数法，因为引用计数解决不了循环引用的问题。

**面试官**：那 GC 具体是怎么回收的呢？用的什么算法？

**候选人**：有四种基础算法。最简单的叫**标记-清除**，先标记所有可达对象，然后清除未标记的，缺点是会产生内存碎片。第二种叫**标记-整理**，标记完之后把存活对象压缩到一端，没有碎片但移动对象耗时。第三种是**复制算法**，把内存分成两块，每次只用一块，GC 时把存活对象复制到另一块然后清空当前块，优点是没有碎片且分配快，但浪费一半空间。

实际上 JVM 用的是**分代收集**——不是一种新算法，而是根据不同代的特点选不同算法。新生代对象朝生夕死，每次 GC 回收 98% 以上，用复制算法特别划算，只需要复制极少数存活对象。老年代对象存活率高，用标记-整理算法避免碎片。

**面试官**：你提到分代，能具体讲讲堆的分代结构吗？

**候选人**：JVM 堆分成新生代和老年代。新生代又分成 Eden 区和两个 Survivor 区（S0 和 S1），默认比例 8:1:1。新对象先分配在 Eden 区，Eden 满了触发 Minor GC。存活的对象复制到一个 Survivor 区，每熬过一次 Minor GC 年龄加 1。达到阈值（默认 15）或者 Survivor 区放不下了，就晋升到老年代。老年代满了会触发 Full GC，清理整个堆。另外还有大对象会直接进老年代，避免在新生代来回复制。

**面试官**：你了解哪些垃圾收集器？生产环境一般用什么？

**候选人**：主流的有六种。Serial 是单线程的，只适合客户端小应用。Parallel 是多线程版本，**JDK 8 的默认收集器**，追求吞吐量，适合批处理场景。CMS 是第一个追求低延迟的收集器，大部分阶段并发执行，但用的是标记-清除算法会产生碎片，JDK 14 已经被移除了。

**G1 是 JDK 9 开始的默认收集器**，它的创新点是把堆分成等大的 Region，新生代和老年代不再是物理连续的，而是 Region 的逻辑集合。回收时以 Region 为单位，优先回收垃圾最多的 Region（所以叫 Garbage First）。最关键的是可以设定期望的停顿时间，G1 会根据历史数据预测每个 Region 的回收耗时，在允许的时间内尽量多回收垃圾最多的 Region。

**ZGC 是最新的超低延迟收集器**，JDK 15 生产可用，JDK 21 支持了分代模式。它通过"着色指针 + 读屏障"的技术，把 GC 停顿控制在了亚毫秒级别，而且停顿时间不会随着堆大小增长。适合金融交易、实时推荐等对延迟要求极高的场景。

**面试官**：那你们项目用的什么 GC？有做过调优吗？

**候选人**：我们用的是 JDK 17，默认的 G1 收集器。遇到过一次问题——线上服务偶发性的 P99 延迟飙高，排查后发现是 Mixed GC 触发太晚，老年代积压了大量对象后一次回收不完，偶尔退化成 Full GC。我们的调优比较简单，把 `InitiatingHeapOccupancyPercent` 从默认的 45 降到 35，让并发标记更早触发，同时把 `MaxGCPauseMillis` 从 200 降到 100。调完之后 Full GC 基本消失了，P99 延迟稳定在 50ms 以内。
