# Agent 如何做意图识别

> 分类：大模型 / AI
> 标签：AI Agent, 意图识别, 意图分类, Function Calling, LLM 应用

## 面试精答

> 适合面试时口述的简洁版本（2-3分钟能说完的要点）

Agent 的意图识别本质上是**把用户的自然语言输入映射到结构化的动作空间**。主流有三种方案：

1. **LLM 直接路由（Prompt-based Classification）**：在 System Prompt 中定义所有意图及其描述，让 LLM 直接输出意图标签。简单直接，适合意图数量少（< 10 个）的场景，但意图多了之后准确率下降。

2. **Function Calling / Tool Use**：将每个意图定义为一个"工具"，LLM 通过选择工具来隐式完成意图识别。这是目前最主流的做法，因为模型在预训练和后训练阶段已经大量接触过 function calling 数据，天然擅长。好处是意图识别和参数抽取一步完成。

3. **分层路由（Cascade Router）**：先粗分类（如"编码类"、"闲聊类"、"工具调用类"），再在子类中细分类。适合意图很多（几十上百个）的复杂 Agent。每一层可以用不同的模型——粗分类可以用轻量模型（快、便宜），细分类用强模型（准）。

**关键经验**：意图识别的准确率往往不取决于模型能力，而取决于**意图定义是否清晰、意图之间是否互斥**。两个意图描述重叠越大，误判率越高。

## 原理深入详解

> 深入底层原理，理解"为什么"

### 方案一：Prompt-based 直接分类

```
System Prompt:
你是一个意图识别器。用户输入会属于以下意图之一：
- QUERY_ORDER: 用户想查询订单状态、物流信息
- REFUND: 用户想申请退款、退货
- COMPLAINT: 用户投诉、表达不满
- CHITCHAT: 闲聊、与业务无关

请直接输出意图标签。

User: 我昨天买的东西怎么还没到
Assistant: QUERY_ORDER
```

**优点**：实现简单，一个 Prompt 搞定。
**缺点**：
- 意图多了之后（>15 个），LLM 容易混淆，准确率显著下降
- 没有结构化的参数抽取能力，需要额外步骤
- 缺乏置信度输出，无法做"低置信度兜底"策略

### 方案二：Function Calling（主流方案）

将意图定义为工具：

```json
{
  "tools": [
    {
      "name": "query_order",
      "description": "查询订单状态、物流进度、配送信息",
      "parameters": {
        "order_id": "订单号",
        "query_type": "查询类型：status | logistics | delivery_time"
      }
    },
    {
      "name": "create_refund",
      "description": "申请退款或退货",
      "parameters": {
        "order_id": "订单号",
        "reason": "退款原因",
        "refund_type": "refund_only | return_and_refund"
      }
    },
    {
      "name": "chitchat",
      "description": "处理与业务无关的闲聊对话",
      "parameters": {
        "response": "回复内容"
      }
    }
  ]
}
```

用户输入 `我昨天买的东西怎么还没到`，模型输出：

```json
{"name": "query_order", "arguments": {"order_id": null, "query_type": "logistics"}}
```

**为什么这是主流方案**：
1. **一步到位**：意图识别 + 参数抽取 + 动作路由全部完成
2. **模型原生支持**：Claude、GPT 等模型在训练时大量接触 function calling 数据
3. **结构化输出**：参数是结构化的 JSON，直接传给后端系统
4. **可扩展**：新增意图只需新增 tool 定义，不需要改 Prompt 逻辑

### 方案三：分层路由（Cascade）

```
用户输入
  ↓
第一层：轻量模型（如 Haiku）粗分类
  ├── 编码类 → 路由到编码 Agent（进一步识别：生成代码 / 修复 bug / 代码审查）
  ├── 知识问答类 → 路由到 RAG Agent
  ├── 工具调用类 → 路由到工具编排 Agent（识别调哪个工具）
  └── 闲聊类 → 直接回复
```

**适用场景**：
- 意图数量 > 20 个
- 不同意图的处理逻辑差异大（有的走 RAG，有的走工具调用）
- 需要控制成本（粗分类用便宜模型，只有需要时才调贵模型）

### 意图定义的核心原则

意图识别的准确率，80% 取决于意图定义的质量：

| 原则 | 说明 | 反例 |
|---|---|---|
| 互斥性 | 两个意图之间不应有重叠地带 | "查询订单"和"查询物流"是同一件事，不应拆开 |
| 完备性 | 所有可能的输入都能归到某个意图 | 没有兜底意图（如 "OTHER"），遇到未知输入就崩 |
| 描述精准 | description 要写清触发条件和边界 | "查询相关"太模糊，"查询订单状态、物流进度、配送时间"更清晰 |
| 颗粒度合理 | 不是越细越好，过细反而容易混淆 | "退款"不需要拆成"退款咨询"、"退款申请"、"退款进度查询" |

### 多意图识别

用户一句话可能包含多个意图：`帮我查一下订单状态，如果不满意我想退款`。

处理方式：
1. **Function Calling 天然支持**：模型可以返回多个 tool_call
2. **串行执行**：先查订单，拿到结果后再判断是否退款
3. **并行执行**：如果两个意图独立，可以同时调用（但需要注意依赖关系）

### 低置信度兜底策略

当模型对意图判断不确定时：

```
LLM 输出 tool_call 的同时输出 confidence（或在 logprobs 中读取）

if confidence < 0.7:
    → 澄清确认："您是想查订单状态，还是想申请退款？"
else:
    → 直接执行
```

实际中，很多 Agent 系统不做显式置信度判断，而是通过 **"无匹配意图时走兜底"** 的方式实现——定义一个 `fallback` 工具，当其他工具都不匹配时模型自然会选择它。

## 常见追问与应对

**Q1: Function Calling 和直接 Prompt 分类，怎么选？**
> 意图少（< 10 个）、不需要参数抽取 → Prompt 分类更简单。意图多或需要结构化参数 → Function Calling 是更好的选择。实际项目中 90% 的场景用 Function Calling，因为它一步完成意图识别 + 参数抽取。

**Q2: 意图识别的准确率能做到多高？**
> 意图定义清晰、互斥性好、数量在 10-20 个时，用 Claude/GPT-4 级别模型，准确率通常能达到 **90-95%**。但意图之间描述重叠大、数量超过 30 个时，准确率会明显下降到 **75-85%**，需要用分层路由或 Few-shot 来补救。

**Q3: 用户输入很模糊怎么办？**
> 三种策略：1）多轮澄清——追问用户确认意图；2）返回最可能的意图 + 确认话术（"您是想查询订单对吗？"）；3）走通用 Agent 路径，让 Agent 自行判断和探索。选择哪种取决于场景容忍度——客服场景优先澄清，助手场景可以容忍一定误判。

**Q4: 意图识别可以离线做吗？还是必须实时？**
> 取决于场景。实时场景（对话、客服）必须在线做。但有些意图可以做**预判**：用户打开退款页面 → 预判意图是退款 → 提前加载退款工具。也可以对历史用户行为做离线意图预测，用于推荐和引导。
