# 开发中的 Token 优化技巧

> 分类：大模型 / AI
> 标签：Token 优化, 成本控制, Prompt 精简, 缓存, 流式输出, LLM 工程化

## 面试精答

> 适合面试时口述的简洁版本（2-3分钟能说完的要点）

Token 优化不是单纯的"少写字"，而是**在保证效果的前提下，用最少的 Token 完成任务**。优化的钱从三个方向省：**输入端少给、模型端少算、输出端少生成**。

**输入端六个技巧**：
1. **Prompt 精简** — 删掉 LLM 已经知道的知识和废话，用短指令替代长描述，一条规则合并多条
2. **System Prompt 缓存（Prompt Caching）** — 不变的 System Prompt + 工具定义做缓存，多轮对话中只重新计算变化的部分
3. **工具返回值裁剪** — 搜索结果只返回 Top-K、日志做摘要、API 返回值只取必要字段
4. **RAG 替代全量文档** — 按需检索相关片段塞进上下文，不把整份文档塞进去
5. **历史对话压缩** — 早期轮次用摘要替代原文，只保留最近 N 轮完整对话
6. **工具定义按需加载** — 不把所有工具定义一次性全部塞进去，根据意图只加载相关工具

**模型端两个技巧**：
7. **混合模型策略** — 简单任务用小模型（GPT-4o-mini / Claude Haiku），复杂任务才用大模型
8. **结构化输出** — 用 JSON Mode 或 `response_format` 强制输出 JSON，省掉模型生成"好的，我来帮你分析"之类的废话

**输出端一个技巧**：
9. **限制最大输出长度** — 设置 `max_tokens`，简单问答不需要 4096 的输出空间

**一句话总结**：Token 优化的本质是**信息密度最大化**——每个 Token 都应该承载有用的信息。

## 原理深入详解

> 深入底层原理，理解"为什么"

### 一、先搞清楚 Token 是什么、怎么计费

#### Token 不是字符

```
LLM 处理文本的最小单位是 Token，不是字符。

同一个词在不同语言中 Token 数差异巨大：

  "Hello world"         → 2 tokens
  "你好世界"             → 4-6 tokens（中文通常 1 个字 = 1-2 tokens）
  "你好世界"（英文翻译）   → 2 tokens

  英文：约 1 token = 4 个字符（平均）
  中文：约 1 token = 1-2 个汉字
  代码：约 1 token = 3-4 个字符

这就是为什么中文 Prompt 比英文贵——同样意思的中文比英文多用 2-3 倍 Token
```

#### 计费方式

```
主流模型的计费方式：

  Claude Sonnet：     $3 / 百万输入 Token,  $15 / 百万输出 Token
  Claude Haiku：      $0.25 / 百万输入 Token, $1.25 / 百万输出 Token
  GPT-4o：           $2.5 / 百万输入 Token,  $10 / 百万输出 Token
  GPT-4o-mini：      $0.15 / 百万输入 Token, $0.6 / 百万输出 Token

两个关键发现：

  1. 输出比输入贵 5 倍
     → 减少输出 Token 的省钱效果是减少输入的 5 倍
     → 阻止模型说废话比精简 Prompt 更省钱

  2. 小模型比大模型便宜 10-20 倍
     → 如果任务简单，用小模型是最大的省钱手段

实际影响：
  假设一个 Agent 每次调用输入 5000 Token、输出 1000 Token
  用 Claude Sonnet：5000 × 3/1M + 1000 × 15/1M = $0.03/次
  日调 10 万次 → $3000/天

  优化后输入 2000 Token、输出 300 Token
  → $0.0105/次 → 日调 10 万次 → $1050/天
  → 每天省 $1950（省了 65%）
```

### 二、技巧一：Prompt 精简

这是**成本最低、见效最快**的优化。不需要改架构，只改文字。

#### 删掉 LLM 已经知道的知识

```
❌ 浪费 Token 的 Prompt（150 Token）：

你是一个非常专业的 Java 开发工程师，拥有 10 年以上的开发经验。
你精通 Java 语言的所有特性，包括但不限于：面向对象编程、
泛型、集合框架、并发编程、JVM 调优等。
你也非常熟悉 Spring Boot、MyBatis、MySQL、Redis 等技术栈。
现在请你帮我分析以下代码的性能问题：

LLM 已经知道 Java、Spring Boot、MySQL 的所有知识
"你很专业"这个描述对输出质量几乎没影响
这些文字纯粹浪费 Token

✅ 精简后（30 Token）：

分析以下代码的性能问题，给出优化建议：
```

#### 用短指令替代长描述

```
❌ 长描述（80 Token）：

当你遇到错误的时候，请按照以下格式输出：
首先输出一行"错误类型："，后面跟上错误的类型名称。
然后输出一行"错误原因："，后面跟上导致错误的原因分析。
最后输出一行"修复建议："，后面跟上你的修复建议。

✅ 短指令（20 Token）：

错误分析格式：
错误类型：{type}
错误原因：{reason}
修复建议：{suggestion}

用"格式模板"替代"步骤描述"，Token 减少且效果更好
因为 LLM 对模板的理解比步骤描述更准确
```

#### 合并重复规则

```
❌ 三条规则（60 Token）：

1. 回答必须准确，不要编造不存在的事实
2. 如果不确定，请明确说"我不确定"而不是猜测
3. 不要凭空捏造数据、引用或案例

✅ 合并成一条（15 Token）：

不确定时如实说明，不要编造事实

三条规则说的是同一件事：不要瞎编。合并成一条。
```

#### 用中文还是英文写 Prompt

```
从 Token 角度：英文更省 Token（同样意思英文 Token 数少 2-3 倍）
从效果角度：大多数模型英文理解略优于中文（训练数据英文更多）

但实际选择取决于：
  - 团队语言环境（全中文团队用英文 Prompt 维护成本高）
  - 模型选择（国产模型中文处理更好）
  - 是否面向用户（如果 Prompt 中有用户可见的部分，用中文）

折中方案：System Prompt 用英文（省 Token + 效果略好），
用户相关的内容用中文。
```

### 三、技巧二：System Prompt 缓存（Prompt Caching）

这是**多轮对话场景中最大的优化手段**。

#### 没有缓存的情况

```
多轮对话中，每次调用 LLM 都要发送完整的上下文：

第 1 轮：
  [System Prompt: 2000 Token] + [用户消息: 100 Token]
  → 输入 2100 Token

第 2 轮：
  [System Prompt: 2000 Token] + [第1轮对话: 300 Token] + [用户消息: 100 Token]
  → 输入 2400 Token

第 3 轮：
  [System Prompt: 2000 Token] + [前2轮对话: 600 Token] + [用户消息: 100 Token]
  → 输入 2700 Token

System Prompt 的 2000 Token 每次都重新计算，完全没变
这是纯粹的浪费
```

#### Prompt Caching 的工作原理

```
Prompt Caching（Anthropic）或 Prompt Caching（OpenAI 叫 Cached Context）：

  原理：LLM 推理时，对输入的 Token 逐个做 Transformer 计算（KV Cache）
        如果前面的 Token 和上一次调用完全一样
        → 直接复用上次计算的 KV Cache
        → 跳过重复计算
        → 只对新增的 Token 做计算

  效果：缓存的 Token 价格降低 90%
        Anthropic：缓存命中 $0.30/百万（原价 $3）→ 省 90%
        延迟也大幅降低（跳过重复计算 = 更快返回）

怎么触发缓存：

  # Anthropic Claude 的做法
  # 在 API 调用中标记哪些 message 做 cache_control
  messages = [
      {
          "role": "system",
          "content": [
              {
                  "type": "text",
                  "text": "很长的 System Prompt...",
                  "cache_control": {"type": "ephemeral"}  # 标记为可缓存
              }
          ]
      },
      # ... 对话历史
  ]

  # OpenAI 的做法（Automatic Caching）
  # 自动缓存：只要前缀相同就自动命中缓存
  # 不需要手动标记
```

#### 缓存的最佳实践

```
最大化缓存命中率的原则：

1. System Prompt 放最前面，且保持不变
   ❌ 每轮动态修改 System Prompt
   ✅ System Prompt 固定，动态信息放在 user message 里

2. 工具定义放 System Prompt 里（因为工具通常不变）
   缓存了 System Prompt 就缓存了工具定义

3. 长文档/参考资料做缓存
   把 RAG 检索到的长文档标记为 cache_control
   同一段文档在多轮对话中只算一次

4. 缓存有 TTL（过期时间）
   Anthropic：5 分钟（300 秒）
   → 如果用户两次对话间隔 > 5 分钟，缓存失效
   → 需要在设计上尽量让连续对话密集发生
```

### 四、技巧三：工具返回值裁剪

Agent 场景中，工具返回值是 Token 膨胀的最大元凶。

```
一次工具调用返回值的对比：

❌ 未裁剪（8000 Token）：
  搜索工具返回 50 条结果，每条 500 字的摘要
  其中只有 3 条和当前任务相关

✅ 裁剪后（800 Token）：
  1. 只返回 Top 5 最相关的结果
  2. 每条摘要截断到 100 字
  3. 用 LLM 对搜索结果做一次预摘要
     "搜索结果概要：共找到 50 条相关结果，
      其中 3 条高度相关，主要涉及..."
  4. 如果 Agent 需要更多细节，再单独调用工具获取

关键思路：不要把工具返回的原始数据直接灌入上下文
         先在工具层面做裁剪，只保留有用的
```

```
工程实现：

def search_tool(query: str, top_k: int = 5, max_length: int = 100):
    """搜索工具：返回裁剪后的结果"""
    raw_results = vector_db.search(query, limit=50)

    # 策略 1：只取 Top-K
    results = raw_results[:top_k]

    # 策略 2：每条截断
    for r in results:
        r["content"] = r["content"][:max_length]

    # 策略 3：添加摘要前缀
    summary = f"共找到 {len(raw_results)} 条结果，展示最相关的 {top_k} 条：\n"

    return summary + format_results(results)
```

### 五、技巧四：RAG 替代全量文档

```
场景：让 LLM 回答关于公司内部文档的问题

❌ 全量加载（100,000 Token）：

  Prompt: "以下是我们公司的完整技术文档（50 万字）：{全部文档内容}
           请回答用户的问题：{question}"

  问题：
    1. 100,000 Token 的输入费用 ≈ $0.3/次
    2. 超出大部分模型的上下文窗口
    3. Lost in the Middle：模型对中间部分的注意力下降

✅ RAG 检索（3,000 Token）：

  1. 用户提问 → 把问题转成向量
  2. 在向量数据库中检索 Top-5 最相关的文档片段
  3. 只把这 5 个片段（约 3000 Token）塞进 Prompt

  Prompt: "以下是相关的文档片段：{5 个片段}
           请基于以上信息回答：{question}"

  效果：100,000 Token → 3,000 Token，省 97%
         且只加载相关内容，模型注意力更集中
```

### 六、技巧五：历史对话压缩

多轮对话中，历史消息只增不减。压缩策略有三种粒度。

```
粒度 1：滑动窗口（最简单）

  只保留最近 N 轮对话，更早的丢弃
  适合：对话中早期信息不重要的场景（闲聊、简单问答）

  实现：messages = messages[-N:]

粒度 2：摘要压缩（最常用）

  把早期对话用 LLM 压缩成一段摘要
  最近 3-5 轮保留完整原文

  触发时机：Token 数超过阈值的 70%

  Token 节省：30 轮完整历史 60K → 摘要 2K + 最近 5 轮 10K = 12K（省 80%）

  （详细实现见 Agent 上下文过长文档）

粒度 3：结构化提取（最精确）

  不是对历史做摘要，而是从中提取结构化信息：

  提取的内容：
    - 用户需求列表
    - 已得出的结论
    - 用户表达的偏好
    - 待解决的问题

  存成 JSON，每轮更新：
    {
      "user_requirements": ["用 Python 实现", "不依赖第三方库"],
      "conclusions": ["选择红黑树而不是 AVL 树"],
      "preferences": ["代码注释用中文"],
      "open_questions": ["如何处理并发插入"]
    }

  Token 占用：结构化 JSON ≈ 200-500 Token
             完整对话历史 ≈ 10000+ Token
```

### 七、技巧六：工具定义按需加载

```
一个 Agent 定义了 20 个工具，每个工具描述 200 Token
→ 工具定义就占了 4000 Token

但用户当前问的是"帮我查一下订单状态"
只需要 1 个查询工具，其他 19 个完全没用

方案：工具按需加载

  意图识别阶段：
    用户输入 → 小模型（Haiku）判断意图 → "订单查询"
    → 只加载 order_query 工具的定义（200 Token）
    → 不加载其他 19 个工具（省 3800 Token）

  两阶段调用：
    第 1 次：小模型做意图识别 + 工具选择（输入少，便宜）
    第 2 次：大模型带着选定的工具执行任务（只带相关工具）
```

### 八、技巧七：混合模型策略

**这是成本优化中效果最显著的单一手段。**

```
核心思路：不是所有任务都需要最强模型

任务分级：
  L0 - 简单任务（意图分类、关键词提取、格式化）
       → Haiku / GPT-4o-mini（成本的 1/10-1/20）

  L1 - 中等任务（总结摘要、工具选择、简单问答）
       → Sonnet / GPT-4o（性价比最优）

  L2 - 复杂任务（代码生成、多步推理、长文写作）
       → Opus / GPT-4（最强能力）

成本对比（同样 1000 Token 输入 + 500 Token 输出）：
  Opus：     $15/M × 1000 + $75/M × 500 = $0.0525
  Sonnet：   $3/M × 1000  + $15/M × 500 = $0.0105
  Haiku：    $0.25/M × 1000 + $1.25/M × 500 = $0.000875

  Haiku 比 Opus 便宜 60 倍！

实际策略：
  - 路由层：先调 Haiku 判断任务复杂度，再决定用哪个模型
  - Plan-and-Execute：规划用 Opus，执行用 Haiku
  - 缓存层：相似问题直接返回缓存，不调模型
```

### 九、技巧八：结构化输出

```
LLM 默认的输出行为：

  用户：帮我查一下北京天气
  LLM：好的，我来帮你查一下北京的天气情况。
       根据最新的气象数据，北京今天的天气是晴天，
       气温 25°C，湿度 45%，东南风 3 级。
       建议出门带一件薄外套，早晚温差较大。

  实际有用信息：天气=晴, 温度=25°C, 湿度=45%, 风力=3级
  "好的我来帮你查一下" → 废话，浪费输出 Token（输出比输入贵 5 倍）

  强制 JSON 输出（节省 40-60% 输出 Token）：

  用户：帮我查一下北京天气，以 JSON 格式返回
  LLM：{"weather":"晴","temp":25,"humidity":45,"wind":"东南风3级"}

  JSON Mode 的实现：
    OpenAI：  response_format={"type": "json_object"}
    Anthropic：在 Prompt 中要求返回 JSON，或用 tool_use 代替

  进一步省 Token：如果输出只是给程序用的（不需要给人看），
  用最短的 key 名：
    {"w":"晴","t":25,"h":45,"wd":"东南风3级"}
```

### 十、技巧九：输出长度限制

```
max_tokens 参数控制模型最多生成多少 Token

默认行为：
  如果不设 max_tokens，模型会一直生成直到"觉得说完了"
  一个简单的事实问答可能被回复成三段论

场景化设置：

  简单问答：max_tokens = 100-200
    "Python 的 list 和 tuple 区别？" → 200 Token 足够

  摘要任务：max_tokens = 300-500
    一段话的摘要不需要 2000 Token

  代码生成：max_tokens = 1000-2000
    视代码长度而定

  长文写作：max_tokens = 4000-8000
    需要完整论述的场景

  设置过短的副作用：
    如果 max_tokens 不够，输出会被截断
    → 信息不完整，用户需要再追问 → 反而多花 Token
    → 所以要根据任务类型设合理的上限
```

### 十一、Token 优化的量化监控

不能凭感觉优化，要有数据。

```
需要监控的指标：

1. 每次 LLM 调用的 Token 消耗
   输入 Token 数、输出 Token 数
   按功能/接口聚合

2. Token 消耗的组成分解
   System Prompt 占多少
   工具定义占多少
   对话历史占多少
   工具返回值占多少
   用户输入占多少

3. 成本追踪
   每日/每周/每月的 API 费用
   按模型、按功能、按用户的费用分布

实现方式：
   每次 LLM 调用后记录 response.usage：
     prompt_tokens, completion_tokens, total_tokens

   发送到监控系统（Prometheus + Grafana / Datadog）

   设置告警：
     - 单次调用 > 10000 Token → 告警（可能有异常大请求）
     - 日费用超过预算 120% → 告警
     - 缓存命中率 < 50% → 告警（缓存策略需要优化）

优化前后对比示例：
  优化前：平均 8000 Token/次，$0.05/次
  优化后：平均 2500 Token/次，$0.015/次
  节省：70% 成本
```

```
按"投入产出比"从高到低排序：

排名  手段                   节省幅度      实现难度
─────────────────────────────────────────────────
 1   混合模型策略             60-95%       中（需要路由层）

 4   Prompt 精简              30-50%       低（改文字）
 5   结构化输出               30-60%       低（参数配置）

```

## 常见追问与应对

**Q1: Token 优化会不会影响模型输出质量？**
> 会，但可以控制。关键原则是"删冗余不删信息"——删掉的是废话和重复，不是有用内容。实际操作中，Prompt 从 2000 Token 精简到 500 Token，输出质量几乎不变（因为被删掉的本来就没用）；工具返回值从 50 条裁剪到 5 条，对当前任务可能反而更好（信息更聚焦）。真正会影响质量的场景是：RAG 检索不准导致漏掉关键片段，或者摘要压缩丢失了重要结论。这些需要用评测集来验证优化前后的质量差异。

**Q2: Prompt Caching 和普通的缓存（比如 Redis 缓存 LLM 的回答）有什么区别？**
> 两个完全不同的东西。Redis 缓存是"同样的输入直接返回之前的输出"——适用于完全相同的问题重复出现（比如 FAQ）。Prompt Caching 缓存的是"LLM 推理过程的中间计算结果（KV Cache）"——即使后面的对话内容变了，前面不变的部分也不用重新计算。打个比方：Redis 缓存是"上次做过这道题，直接抄答案"；Prompt Caching 是"上次做到第 3 步，这次从第 3 步继续做"。

**Q3: 混合模型策略的路由层怎么做？不会引入额外延迟吗？**
> 路由层本身就是一次轻量级 LLM 调用，通常用最小最便宜的模型（Haiku）。输入只有用户消息 + 分类 Prompt，约 100-200 Token，延迟 50-100ms。和它省下的成本相比（把 90% 的调用从 Opus 降到 Haiku），这点延迟完全可以接受。实际还可以更优化：用关键词匹配做快速路由（"翻译"→ 翻译模型，"代码"→ 代码模型），只有不确定的才调 LLM 做分类。

**Q4: 怎么知道 Prompt 里哪些内容是"废话"可以删？**
> 最直接的方法：A/B 测试。把 Prompt 精简前后的版本分别跑同一批测试问题，对比输出质量。如果质量不变甚至变好，说明删掉的内容确实是废话。更工程化的做法：用评测集做自动化回归——维护 50-100 个标准测试用例，每次修改 Prompt 都跑一遍，确保没有质量退化。

**Q5: 中文比英文贵 2-3 倍 Token，那中文场景应该用英文 Prompt 吗？**
> 不一定。要权衡三个因素：1）**Token 成本**——英文确实更省；2）**输出质量**——对于中文问答场景，中文 Prompt 的理解准确度通常更高（用户问中文问题，Prompt 也是中文，模型更不容易理解偏差）；3）**维护成本**——全中文团队的代码库里混入大量英文 Prompt，维护难度增加。实际建议：System Prompt（不给用户看的）用英文省 Token，用户可见的交互内容用中文保证体验。

**Q6: 这些优化技巧只适用于 Agent 场景吗？**
> 不是，所有 LLM 应用都适用。分类一下：**所有场景通用**的——Prompt 精简、结构化输出、max_tokens、混合模型；**多轮对话场景**的——Prompt Caching、历史压缩；**Agent 场景**的——工具返回值裁剪、工具按需加载；**文档问答场景**的——RAG 替代全量文档。可以根据自己的应用类型选择性实施。

## 完整面试回答

> 模拟面试中的实际对话，展示如何从头到尾回答 Token 优化问题

**面试官**：在实际开发 LLM 应用的时候，你做过哪些 Token 优化的工作？

**候选人**：做过不少。Token 优化的核心思路是三个方向——**输入端少给、模型端少算、输出端少生成**。

输入端最有效的是 **Prompt 精简**和 **Prompt Caching**。Prompt 精简就是删掉废话——LLM 已经知道的知识不用写、"你很专业"这种描述对输出没帮助还浪费 Token。我们做过统计，精简前后 Prompt 从 2000 Token 降到 500 Token，输出质量几乎不变。Prompt Caching 更厉害，多轮对话中 System Prompt 和工具定义是不变的，这部分做缓存后，重复计算的 Token 费用降低 90%。

Agent 场景中**工具返回值裁剪**特别重要。搜索工具一次返回 50 条结果就是几万 Token，我们做了 Top-K 过滤 + 每条截断 + 相似合并，从 8000 Token 降到 800 Token。另外还做了**工具按需加载**——先让小模型做意图识别判断需要哪些工具，只加载相关的工具定义，不把 20 个工具全塞进去。

**面试官**：模型选择上有什么策略？

**候选人**：**混合模型策略是成本优化中效果最显著的单一手段**。我们把任务分级：简单的意图分类、关键词提取用 Haiku，成本只有 Opus 的 1/60；中等任务用 Sonnet；复杂的多步推理才用 Opus。具体实现是加一个路由层——用户请求进来先调 Haiku 判断复杂度，再决定用哪个模型。引入路由层只多了 50-100ms 延迟，但整体成本降了 80% 以上。

**面试官**：怎么验证优化后质量没有下降？

**候选人**：我们维护了一个评测集，大概 80 个标准测试用例，覆盖各种场景。每次修改 Prompt 或调整策略后都跑一遍评测集对比优化前后的输出质量。另外也有实时监控——记录每次 LLM 调用的输入输出 Token 数、延迟、费用，按功能聚合。如果某次优化后评测集质量下降超过 2%，就回滚重新调整。
