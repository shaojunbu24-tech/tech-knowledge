# AI Agent 的调优评估与评测集构建

> 分类：大模型 / AI
> 标签：AI Agent, 评估, 评测集, 调优, LLM 应用, Case Study

## 面试精答

> 适合面试时口述的简洁版本（2-3分钟能说完的要点）

Agent 调优评估分三个层次：**离线评测（评测集打分）→ 在线评测（沙箱/影子模式）→ 线上评测（A/B 测试）**。评测集是离线评测的基础，决定了调优方向是否正确。

### 评测维度

一个 Agent 的评估至少覆盖四个维度：

1. **任务完成率（Task Completion Rate）**：最终结果是否正确，是最核心的指标
2. **工具调用准确率（Tool Use Accuracy）**：是否选对了工具、传对了参数、调用顺序是否合理
3. **效率指标**：平均步数（步数越少越高效）、Token 消耗、首 Token 延迟
4. **鲁棒性**：面对边界 case、异常输入、工具报错时的恢复能力

### 评测集怎么构建

评测集不是随便找几条 prompt，而是有严格的方法论：

1. **从线上日志采样**：收集真实用户请求，按任务类型分层采样（高频场景多采，长尾场景覆盖到）
2. **人工标注 Golden Trajectory**：对每条输入，人工标注"理想的工具调用链 + 最终正确答案"
3. **难度分级**：简单（单步可完成）、中等（需要 2-3 步推理 + 工具调用）、困难（多步推理 + 容错 + 异常处理）
4. **按场景分类**：编码类、数据分析类、信息检索类、多轮对话类等，确保覆盖面
5. **持续迭代**：线上新出现的 Bad Case 及时补充到评测集中

评测集规模通常在 **200-2000 条**之间，太少不具统计意义，太多标注成本过高。

## 原理深入详解

> 深入底层原理，理解"为什么"

### 评测集构建的完整流程

```
阶段一：数据采集
  线上日志 → 去重 → 去隐私 → 按场景分层采样 → 得到原始 query 集

阶段二：人工标注
  原始 query → 标注师标注 ideal trajectory → 质检（交叉标注 + 一致性校验）→ 得到评测集 v1

阶段三：评测执行
  评测集 → Agent 执行 → 收集实际 trajectory → 与 golden trajectory 对比 → 计算各维度指标

阶段四：持续维护
  线上 Bad Case → 补充到评测集 → 重新评测 → 验证修复效果
```

### 评测指标的具体计算

| 指标 | 计算方式 | 说明 |
|---|---|---|
| 任务完成率 | 正确完成的任务数 / 总任务数 | 需要明确定义"正确"：精确匹配 or 语义相似度阈值 |
| 工具调用 F1 | 对比预测的工具调用序列与 golden 序列 | 精确率（调用的都该调）+ 召回率（该调的都调了） |
| 平均步数 | Agent 完成任务的平均 action 数 | 过多说明效率低或存在循环 |
| Token 效率 | 完成 rate / 总 token 消耗 | 衡量性价比 |
| 鲁棒性得分 | 异常 case 下的完成率 | 工具返回错误时能否自行恢复 |

### 调优的主要手段

1. **Prompt Engineering**：调整 System Prompt 中的角色设定、工具描述、推理约束（如"先思考再调用工具"）
2. **Few-shot Examples**：在 Prompt 中加入高质量示例 trajectory，引导 Agent 模仿正确行为
3. **工具描述优化**：改写工具的 name、description、parameters 描述，让 Agent 更容易选对工具
4. **SFT（监督微调）**：用标注好的 trajectory 数据对模型做微调，让模型内化正确的工具使用模式
5. **RLHF / DPO**：对 Agent 的多种 trajectory 做偏好标注，用强化学习优化策略

### 自动化评测框架

业界常用的自动化评测方案：

- **LLM-as-Judge**：用强模型（如 GPT-4 / Claude）作为裁判，对 Agent 输出打分，适合语义类评估
- **规则匹配**：对工具调用序列做精确匹配或正则匹配，适合结构化评估
- **环境模拟（Sandbox）**：在沙箱环境中运行 Agent，验证执行结果（如数据库查询结果是否正确）
- **组合方案**：规则匹配评估工具调用 + LLM-as-Judge 评估最终回答质量

## 常见追问与应对

**Q1: LLM-as-Judge 有什么局限性？怎么缓解？**
> 局限：1）裁判模型本身有偏差，倾向给更长的回答更高分；2）对工具调用的正确性判断不如规则精确；3）成本高。缓解：关键指标用规则匹配兜底，LLM-as-Judge 只评估语义质量；多裁判投票减少单模型偏差；定期人工抽检校准。

**Q2: 评测集会不会"过拟合"？怎么保证泛化能力？**
> 会。如果反复用同一套评测集调 Prompt，Agent 会针对评测集优化而非真正提升能力。缓解方法：1）评测集分 train/holdout 两份，调优只看 train，最终验证用 holdout；2）定期用线上真实流量做 A/B 测试验证；3）评测集持续更新，加入新的 case 类型。

**Q3: 调优一个 Agent 通常要多久？瓶颈在哪？**
> 取决于复杂度。简单 Agent（2-3 个工具）1-2 周；复杂 Agent（10+ 工具、多轮推理）1-3 个月。瓶颈通常在：1）评测集标注质量（需要领域专家）；2）Bad Case 分析（每个失败 case 都要人工归因）；3）工具描述的迭代（往往改一个词就能大幅提升工具选择准确率）。

**Q4: SFT 和 Prompt Engineering 分别适合什么场景？**
> Prompt Engineering 适合快速迭代、工具数量少、场景明确的 Agent，成本低、见效快。SFT 适合：1）Prompt 已经塞不下了（工具太多）；2）需要模型内化复杂推理链路；3）对延迟有要求（Few-shot 增加了 Prompt 长度和推理时间）。实际中通常先用 Prompt Engineering 摸到天花板，再用 SFT 突破。

## 调优成功案例

### Case 1：数据分析 Agent — 工具描述优化带来 40% 的准确率提升

**背景**：一个数据分析 Agent，支持 Text-to-SQL 查询、图表生成、数据解释三个工具。初始版本任务完成率只有 55%，大量错误集中在"工具选择错误"——用户想查数据，Agent 却调用了图表生成工具。

**问题归因**：分析 120 条 Bad Case 后发现，70% 的错误是因为工具描述不够区分。三个工具的 description 都包含"数据"这个关键词，Agent 无法区分。

**调优手段**：
1. 重写工具描述：Text-to-SQL 工具强调"查询、过滤、聚合数据"；图表生成工具强调"可视化、绘制图表"；数据解释工具强调"分析趋势、总结结论"
2. 在 System Prompt 中加入决策树引导："用户要查具体数值 → 调 SQL 工具；要画图 → 调图表工具；要解读 → 调解释工具"
3. 增加 5 条 Few-shot 示例覆盖易混淆场景

**效果**：任务完成率从 55% 提升到 77%，工具调用准确率从 62% 提升到 89%。整个过程只改了 Prompt，没有做任何模型微调。

### Case 2：客服 Agent — SFT 微调解决多轮对话中的状态丢失

**背景**：一个售后客服 Agent，支持查订单、退款、换货、物流追踪四个工具。Prompt 版本在单轮对话中表现良好（完成率 82%），但多轮对话场景（用户需要多次追问、修改需求）完成率暴跌到 48%。

**问题归因**：多轮场景下，Agent 在第 3-4 轮开始"遗忘"上下文，重复调用已调用过的工具，或丢失用户之前提供的关键信息（如订单号）。

**调优手段**：
1. 从线上日志中筛选 500 条多轮对话（含成功和失败），人工标注 ideal trajectory
2. 用这 500 条数据对模型做 SFT 微调，训练目标是让模型在多轮场景下正确维护状态
3. 在 System Prompt 中加入"对话摘要"机制：每轮结束前，Agent 先输出当前状态摘要（用户意图、已获取的信息、待执行的操作）

**效果**：多轮对话完成率从 48% 提升到 79%，单轮完成率从 82% 微升到 85%。上线后用户满意度（CSAT）提升 23%，转人工率下降 31%。
