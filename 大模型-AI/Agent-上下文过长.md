# Agent 上下文过长怎么处理

> 分类：大模型 / AI
> 标签：Agent, 上下文管理, 长上下文, 摘要压缩, RAG, 滑动窗口, Token 优化

## 面试精答

> 适合面试时口述的简洁版本（2-3分钟能说完的要点）

Agent 上下文过长的本质是**信息进多出少，积压在有限的窗口里**。解决思路不是单一的，而是**从源头减少、中间压缩、结尾兜底**三管齐下。

**六个核心手段**：

1. **工具返回值裁剪** — 在源头控制信息量。工具（如日志搜索、数据库查询）返回的结果做截断、摘要、Top-K 过滤，不让原始全量数据灌入上下文
2. **滑动窗口 + 摘要** — 保留最近 N 轮对话，把更早的历史用 LLM 压缩成摘要，用摘要替代原文
3. **RAG 替代全量加载** — 不把所有参考资料塞进上下文，而是按需检索相关的片段，只加载有用的
4. **分层记忆架构** — 短期记忆（当前对话的完整历史）、工作记忆（当前任务的摘要）、长期记忆（跨对话的结构化知识），各层按需加载
5. **子 Agent 分工** — 把复杂任务拆给多个子 Agent，每个子 Agent 维护自己的短上下文，主 Agent 只看各子 Agent 的结论摘要
6. **Prompt 精简** — 压缩 System Prompt、用更短的指令替代冗长描述、去掉重复信息

**一句话总结**：不要问"上下文装不下怎么办"，而要在设计时就控制好"什么该进上下文、什么该放外面、什么该丢弃"。

## 原理深入详解

> 深入底层原理，理解"为什么"

### 一、先搞清楚上下文里都装了什么

```
一个 Agent 的上下文组成：

┌─────────────────────────────────────────────┐
│  System Prompt         ~500-2000 Token      │  ← 固定开销
│  （角色设定、工具描述、规则约束）               │
├─────────────────────────────────────────────┤
│  工具定义               ~1000-5000 Token     │  ← 固定开销
│  （每个工具的 name + description + params）   │     工具越多越大
├─────────────────────────────────────────────┤
│  对话历史               ~5000-50000 Token    │  ← 主要增长源
│  （每轮的 user + assistant + tool_call）      │     越聊越长
├─────────────────────────────────────────────┤
│  工具返回结果           ~2000-20000 Token     │  ← 主要增长源
│  （搜索结果、查询结果、API 返回）              │     不加控制会爆炸
├─────────────────────────────────────────────┤
│  用户当前输入           ~100-1000 Token       │
├─────────────────────────────────────────────┤
│  预留给模型输出         ~2000-4000 Token      │
└─────────────────────────────────────────────┘
  总计可能：30000 - 80000+ Token
```

**上下文膨胀的三大元凶**：
1. **对话历史逐轮累积** — 每轮增加 user + assistant + tool_call + tool_result，10 轮就可能 2 万 Token
2. **工具返回结果过大** — 一次日志搜索返回 200 条日志，每条 200 字，就是 4 万 Token
3. **工具定义过多** — 定义了 20 个工具，每个工具描述 200 Token，光工具定义就 4000 Token

### 二、手段一：工具返回值裁剪（源头控制）

这是**投入产出比最高**的手段——在信息进入上下文之前就控制好量。

#### 返回值截断

```
工具内部实现：

def search_logs(query, limit=50):
    results = elasticsearch.search(query)

    # 裁剪策略 1：限制返回条数
    results = results[:limit]

    # 裁剪策略 2：每条截断到最大长度
    for r in results:
        r["message"] = r["message"][:200]  # 最多 200 字符
        # 去掉堆栈信息中的重复行
        r["stacktrace"] = deduplicate_stacktrace(r["stacktrace"])

    # 裁剪策略 3：合并相同模式的日志
    results = merge_similar(results)
    # "该错误重复出现 156 次" → 一条代替 156 条

    return results
```

#### 摘要式返回

```
不要返回原始数据，返回统计摘要：

❌ 原始返回（2000 Token）：
  [10:00:01] ERROR order-service - timeout
  [10:00:02] ERROR order-service - timeout
  [10:00:03] ERROR order-service - timeout
  ... 共 200 条

✅ 摘要返回（100 Token）：
  错误概览：order-service 在 10:00-10:05 期间
  出现 200 次 timeout 错误。
  错误趋势：10:00 开始，10:02 达到峰值（80次/分钟），
  10:04 开始下降。
  相关服务：MySQL CPU 使用率同期飙升至 95%。

Agent 看到摘要后，如果需要看具体某几条原始日志，
再单独调用工具获取（按需加载）。
```

#### 分页式获取

```
工具设计：

search_logs(query, page=1, page_size=10)
  → 先返回 10 条 + 总数信息
  → "共找到 156 条结果，当前显示第 1-10 条"
  → Agent 看完觉得需要更多，再调 page=2

避免一次性返回全量数据。
```

### 三、手段二：滑动窗口 + 摘要压缩

#### 基本思路

```
上下文窗口 200K Token，对话已进行到 15 轮（60K Token）：

方案 A：滑窗丢弃
  只保留最近 10 轮，更早的直接丢弃
  简单粗暴，可能丢失重要信息

方案 B：摘要压缩
  把前 10 轮压缩成一段摘要（从 40K → 2K Token）
  再保留最近 5 轮的完整历史
  保留了关键信息，大幅减少 Token 占用

实际效果：
  原始：15 轮完整历史 = 60K Token
  压缩后：摘要(2K) + 最近 5 轮(20K) = 22K Token
  节省了 63%
```

#### 摘要的实现

```python
def compress_history(messages, keep_recent_n=5):
    """把早期的对话历史压缩成摘要"""

    if len(messages) <= keep_recent_n:
        return messages  # 不需要压缩

    # 分割：早期消息 vs 最近消息
    old_messages = messages[:-keep_recent_n]
    recent_messages = messages[-keep_recent_n:]

    # 用 LLM 对早期消息做摘要
    summary_prompt = f"""
    请将以下对话历史压缩成一段简洁的摘要，保留：
    1. 用户的核心需求是什么
    2. 已经完成了哪些分析步骤
    3. 得出了什么结论
    4. 用户表达了什么偏好或反馈

    对话历史：
    {format_messages(old_messages)}
    """

    summary = llm.generate(summary_prompt)

    # 摘要作为一条特殊的 system 消息
    compressed = [
        {"role": "system", "content": f"[对话历史摘要]\n{summary}"},
        *recent_messages
    ]

    return compressed
```

#### 摘要的触发时机

```
触发条件（满足任一即触发）：

1. Token 计数触发
   每轮结束后计算当前上下文 Token 数
   超过阈值的 70% → 触发压缩
   阈值 = 上下文窗口 - 输出预留

2. 轮数触发
   对话超过 N 轮（如 10 轮）→ 触发压缩

3. 关键节点触发
   Agent 完成一个子任务后 → 把该子任务的对话压缩成结论
   开始新子任务时 → 只带结论，不带过程
```

#### 多级压缩

```
对话进行很久的情况（30+ 轮）：

Level 0（原始）：最近 5 轮完整历史
Level 1（摘要1）：第 6-15 轮的摘要（~500 Token）
Level 2（摘要2）：第 16-25 轮的更粗粒度摘要（~200 Token）
Level 3（摘要3）：第 26 轮之前的极简摘要（~100 Token）

组装进上下文时：
  [极简摘要3] → [粗粒度摘要2] → [摘要1] → [最近5轮完整]
  越近的信息越详细，越早的信息越压缩
```

### 四、手段三：RAG 替代全量加载

**核心思路**：不把所有参考资料塞进上下文，而是**按需检索只加载相关的**。

```
❌ 不用 RAG（全量加载）：

Agent 要分析日志 → 把整个日志文件（5MB）塞进 Prompt
  → 5MB ≈ 几十万 Token，远超上下文窗口

✅ 用 RAG（按需检索）：

1. 日志文件预先做 Embedding，存入向量数据库
2. Agent 根据当前任务生成查询向量
3. 只检索最相关的 Top-K 片段（比如 5 段，每段 500 字）
4. 把这 5 段塞进上下文（共 ~2500 Token）
5. 不相关的日志完全不加载

节省：几十万 Token → 几千 Token
```

**RAG 在 Agent 中的典型应用**：

```
场景：一个帮助开发者查文档的 Agent

不装全部文档（太大了），而是：
  1. 所有文档做 Embedding 存入向量库
  2. 用户问"Redis 怎么设置过期时间"
  3. 检索到 3 段相关的文档片段
  4. 只把这 3 段塞进上下文
  5. Agent 基于这 3 段回答
```

### 五、手段四：分层记忆架构

把信息按"需要程度"分层，按需加载。

```
┌─────────────────────────────────────┐
│  L0：当前上下文（始终加载）           │
│  - System Prompt                    │
│  - 工具定义                          │
│  - 最近 N 轮对话                     │
│  - 当前任务相关的检索结果             │
├─────────────────────────────────────┤
│  L1：工作记忆（按任务加载）           │
│  - 当前任务的进度和中间结论           │
│  - 与当前任务相关的经验教训           │
│  - 当前对话的摘要                    │
│  → 任务切换时加载/卸载               │
├─────────────────────────────────────┤
│  L2：用户画像（对话开始时加载）       │
│  - 用户偏好                          │
│  - 用户技术栈                        │
│  - 历史反馈规则                      │
│  → 每次对话开始加载一次               │
├─────────────────────────────────────┤
│  L3：长期知识（按需检索）             │
│  - 知识库（RAG）                     │
│  - 历史对话档案                      │
│  - 公司内部文档                      │
│  → 只在需要时检索相关片段             │
└─────────────────────────────────────┘

关键原则：不是所有层都同时装进上下文
L0 始终在，L1/L2/L3 按需加载
```

### 六、手段五：子 Agent 分工

**核心思路**：一个 Agent 上下文不够？拆成多个 Agent，每个维护自己的短上下文。

```
场景：分析一个复杂的系统故障

主 Agent（编排者）：
  上下文：任务描述 + 各子 Agent 的结论摘要
  不包含任何原始日志或指标数据

  ├── 子 Agent 1（日志分析）
  │   上下文：只有日志搜索工具 + 日志片段
  │   输出："发现 order-service 在 10:00-10:05 有 200 次 timeout"
  │
  ├── 子 Agent 2（指标分析）
  │   上下文：只有指标查询工具 + 指标数据
  │   输出："MySQL CPU 10:00 起飙升至 95%，慢查询数同步增长"
  │
  └── 子 Agent 3（变更分析）
      上下文：只有变更查询工具 + 变更记录
      输出："09:58 order-service 发版，新接口 SQL 缺少索引"

主 Agent 汇总三个摘要：
  "根因：09:58 发版引入慢查询 → MySQL CPU 打满 → 服务超时"

上下文消耗：
  单 Agent 全量：日志(20K) + 指标(10K) + 变更(5K) + 历史(15K) = 50K
  多 Agent：主 Agent(~5K) + 子 Agent 各(~10K) = 35K，且每个都不超限
```

**什么时候该拆**：
- 任务天然可分解（日志分析、指标查询、变更查检可以并行）
- 单 Agent 上下文已经接近上限
- 不同子任务需要不同的工具集

### 七、手段六：Prompt 精简

不要忽视 System Prompt 本身的优化。

```
❌ 冗长的 System Prompt（2000 Token）：

你是一个非常专业的日志分析助手。你的名字叫 LogAnalyzer。
你拥有丰富的系统运维经验，擅长分析各种类型的日志，
包括应用日志、系统日志、访问日志、错误日志等。
你会按照以下步骤进行分析：
第一步，先用 aggregate_logs 工具了解全局状况...
第二步，再用 search_logs 工具查看具体错误...
第三步，用 query_metrics 查相关指标...
（后面还有大段重复工具用法的描述）

✅ 精简后的 System Prompt（500 Token）：

你是日志分析助手。

流程：aggregate_logs（全局概览）→ search_logs（具体错误）
     → query_metrics（系统指标）→ query_changes（变更记录）

输出格式：
### 根因 | ### 影响 | ### 建议（紧急/短期/长期）

规则：基于数据下结论，不足时主动调更多工具。

节省 1500 Token，效果几乎相同。
```

**精简技巧**：
1. 删掉 LLM 已经知道的知识（"什么是 TCP"、"什么是 HTTP"）
2. 删掉重复的工具使用说明（工具定义的 description 已经说了）
3. 用列表替代段落（列表更短，LLM 理解更好）
4. 合并相似规则（三条规则合并成一条通用规则）

### 八、Token 预算管理（工程实践）

在生产环境中，给每次 Agent 调用设定 Token 预算。

```
Token 预算分配（假设总预算 128K Token）：

  System Prompt：      ≤ 1,000 Token（精简版）
  工具定义：            ≤ 3,000 Token（控制工具数量）
  用户画像记忆：        ≤ 500 Token
  对话历史摘要：        ≤ 2,000 Token
  最近对话（完整）：     ≤ 10,000 Token（最近 3-5 轮）
  工具返回结果：        ≤ 5,000 Token（每次调用）
  用户当前输入：        ≤ 1,000 Token
  ─────────────────────────────
  已用：               ≤ 22,500 Token
  预留给输出：          ≤ 4,000 Token
  ─────────────────────────────
  安全余量：            ~100,000 Token

监控：
  每轮结束后计算当前 Token 使用量
  超过预算 70% → 触发摘要压缩
  超过预算 90% → 强制摘要 + 丢弃最旧的历史
```

## 常见追问与应对

**Q1: 上下文窗口已经 200K 了，还需要这些优化吗？**
> 需要。三个原因：1）200K 是上限，不代表每次都该用满——Token 费用和延迟随输入增长线性增加，填满 200K 的输入要花好几秒和几毛钱；2）Lost in the Middle 问题——研究表明 LLM 对上下文中间部分的信息"注意力"明显下降，塞太多反而降低回答质量；3）实际场景中工具返回 + 多轮对话很容易超过 200K（一次日志搜索 50K，10 轮对话 30K，已经 80K）。

**Q2: 摘要压缩会不会丢失关键信息？**
> 会，但可以缓解：1）摘要 Prompt 明确列出"必须保留什么"（用户需求、已得出的结论、未解决的问题）；2）重要信息（如用户的明确指令）单独提取存入结构化记忆，不依赖摘要；3）保留最近 N 轮完整历史不压缩，只压缩早期的；4）如果 Agent 需要早期细节，可以让它主动调工具重新获取。

**Q3: 多个子 Agent 和单个 Agent 多次调用有什么区别？**
> 单 Agent 多次调用是串行的，共享同一个上下文，信息会累积。多子 Agent 是并行的，每个有独立上下文，互不干扰。子 Agent 模式适合：1）子任务可以并行（日志分析 + 指标查询同时做）；2）不同子任务需要不同的工具集（减少无关工具定义的 Token 浪费）；3）单 Agent 上下文确实装不下。

**Q4: 怎么监控 Agent 的上下文使用情况？**
> 三层监控：1）**每轮后统计**：记录每轮的输入 Token 数、输出 Token 数、工具返回 Token 数；2）**趋势告警**：如果连续 3 轮 Token 增速加快，提前触发压缩；3）**异常检测**：如果单次工具返回超过 5000 Token，告警并自动截断。用 Prometheus + Grafana 做可视化，Token 使用量就像 CPU 使用率一样监控。
