# ReAct、Plan-and-Execute、Reflection 三种范式核心区别与选型

> 分类：大模型 / AI
> 标签：AI Agent, ReAct, Plan-and-Execute, Reflection, Reflexion, 设计范式, 推理模式

## 面试精答

> 适合面试时口述的简洁版本（2-3分钟能说完的要点）

三种范式解决的是**不同层次的问题**，不是简单的谁比谁好。

**ReAct（边想边干）**：核心循环是「思考→行动→观察→再思考」，没有提前定死的完整计划，走一步看一步，灵活度最高。适合流程不固定、复杂度适中的任务。短板是长流程任务容易跑偏，陷入无效循环。

**Plan-and-Execute（先规划再执行）**：针对 ReAct 长任务跑偏的痛点做的优化。把「规划」和「执行」完全解耦——用一个 LLM 先把大目标拆成完整执行清单，再用另一个模块按清单逐步执行。优势是全局结构清晰、不会跑偏、可并行优化。还可以用「强模型规划、弱模型执行」策略降低 70%-90% 成本。

**Reflection（反思迭代）**：**它不是独立的完整流程**，而是给前两者叠加的「质量增强 buff」。在原有流程上加「生成→评估→改进」闭环，不达标就重试或调整。代价是 token 消耗增加 30%-100%。

**一句话选型**：任务简单用 ReAct，流程长且复杂用 Plan-and-Execute，输出要求高再加 Reflection。别过度工程化，够用就好。

## 原理深入详解

> 深入底层原理，理解"为什么"

### 先搞清两个基础概念

**设计范式** vs **推理模式**——这是理解三种范式的地基：

- **设计范式**：Agent 的「顶层做事流程框架」，就像公司的管理制度，决定整个系统从头到尾按什么大逻辑跑。ReAct、Plan-and-Execute 就是设计范式。
- **推理模式**：Agent 在每一步干活时「脑子里怎么思考的」，就像员工的干活方法，决定每一步的决策怎么来。CoT、Self-Consistency 就是推理模式。

两者关系：**设计范式是公司的管理制度，推理模式是员工的干活方法**，可以自由组合。

### 一、ReAct：单步迭代范式

ReAct（Reasoning + Acting）是所有 Agent 范式的基础。核心循环：

```
思考（Thought）→ 行动（Action）→ 观察（Observation）→ 思考 → ...
```

**生活化理解**：ReAct 就像外卖骑手。接了「送餐到用户手里」的目标，不会提前规划全程路线，而是每一步都根据实际情况实时决策——取餐、导航到小区、选西门更近、找到 3 号楼、送达。中途路封了就立刻改路线。

**核心特点**：
- 规划和执行**混在一起**，没有独立的规划环节
- 每一步的行动完全基于上一步的结果实时调整
- 灵活度最高，但长任务容易遗忘初始目标

**ReAct vs 另外两者**：
- vs Plan-and-Execute：没有「提前做完整规划」的环节
- vs Reflection：循环里没有专门的自我检查环节，只有行动后的结果观察

**Token 消耗分析**（以 5 步任务为例）：

每次调用 LLM 都要把完整历史带上，输入 token 逐轮递增：

```
第 1 步输入: 2,000 token
第 2 步输入: 4,000 token
第 3 步输入: 6,000 token
第 4 步输入: 8,000 token
第 5 步输入: 10,000 token
总计输入: 30,000 token（线性增长）
```

**适用场景**：信息搜索、简单问答助手、客服机器人、流程不固定的中等复杂度任务。也是新手入门首选。

### 二、Plan-and-Execute：规划执行范式

针对 ReAct「长任务容易跑偏」的核心痛点做的针对性优化。

**核心区别一句话**：ReAct 是「走一步看一步，边想边干」；Plan-and-Execute 是「先把完整计划定好，再按计划一步步干」。

**架构拆解**：

```
规划模块（Planner LLM）
  输入：用户目标 + 可用工具列表
  输出：完整的执行步骤清单 [Step1, Step2, Step3, ...]
       ↓
执行模块（Executor）
  逐个步骤执行，每步输出结果
  无依赖的步骤可并行
       ↓
汇总模块
  综合所有步骤结果，生成最终输出
```

**生活化理解**：像项目经理做新产品——先做需求调研、再出原型设计、交给开发、测试验收、最后上线。先定好全流程计划，再按计划推进，不会中途随便改方向。

**隐藏优势：混合模型策略**

```
规划阶段：调 GPT-4 / Claude（推理能力强，确保拆分质量）→ 只调 1 次
执行阶段：调 GPT-4o-mini / 开源 7B 模型（任务已具体化，便宜模型够用）→ 调多次
→ 总成本降低 70%-90%，质量几乎不受影响
```

ReAct 做不到这种差异化分配——每一步都是同一个模型又推理又执行。

**Token 消耗分析**（同样 5 步任务）：

```
规划阶段: ~3,000 token（调 1 次）
执行阶段: ~7,500 token（5 步 × 1,500 token，只带步骤指令 + 结果摘要）
汇总阶段: ~4,000 token（调 1 次）
总计: ~14,500 token → 比 ReAct 的 30,000 低了一半多
```

如果再用混合模型策略，实际花费还能再降 70%。

**适用场景**：写竞品分析报告、全流程项目开发、多维度行业调研、流程长且步骤多的复杂任务。

### 三、Reflection：反思迭代范式

**最关键的一点**：Reflection **不是独立的完整流程**，而是给 ReAct 或 Plan-and-Execute 叠加的增强机制。

**生活化理解**：
- ReAct = 一道题一道题挨着做，做完不回头看
- Plan-and-Execute = 先定好整张卷子的做题顺序和时间分配，再按计划做题
- Reflection = 做完题后回头检查一遍，发现错了马上改

**核心循环**：

```
生成（Generate）→ 评估（Evaluate）→ 改进（Improve）→ 再评估 → ...
```

在原有范式的基础上，加了一个独立的检查环节：
1. 判断当前输出是否达标（设定明确的质量标准）
2. 不达标 → 分析问题原因，调整策略，重新生成
3. 达标 → 输出最终结果

**Token 消耗影响**：每个反思节点至少多一次 LLM 调用，整体 token 消耗在基础范式上增加 **30%-100%**（取决于反思轮次和严格程度）。一般设置最多反思 2-3 轮，避免「为了改而改」的死循环。

**适用场景**：写生产环境代码、正式商业报告、法律文书——凡是有错误就会出大问题的场景。

### 进阶：动态 Replan 和 Reflexion

#### 动态 Replan

解决 Plan-and-Execute「计划定死、中途遇到意外怎么办」的痛点。

```
每步执行完之后:
  → 把当前结果 + 剩余计划交给规划模块
  → 规划模块判断：「原来的计划还合理吗？」
  → 不合理 → 生成新的剩余步骤计划，替换原计划
  → 合理 → 继续执行
```

**效果**：保留「先规划再执行」的结构优势，又不会因计划僵硬而翻车。代价是每步多一次 LLM 调用来评估计划。

#### Reflexion

把 Reflection 推到更深层次——不仅检查输出对不对，还把每次失败的原因总结成「经验教训」存进记忆。

```
普通 Reflection: 做完检查一遍，发现问题就重做
Reflexion:       做完检查 → 分析失败原因 → 写入"错题本" → 下次遇到类似任务先翻错题本

就像做错数学题不是只改答案，而是写上「这类题容易漏掉符号变化，下次特别注意」
```

**实证效果**：在 HumanEval 代码生成基准上，Reflexion 把 GPT-4 的 pass@1 从 **80% 提升到 91%**，提升超过 10 个百分点。同样的基座模型，仅仅加了「反思 + 记住教训」就大幅提高成功率。

**为什么代码生成特别适合 Reflexion**：代码可以运行、可以测试，执行结果就是最直接的反馈信号。Agent 写完代码跑一遍测试，没通过就分析问题，把「这个 API 参数顺序搞反了」「边界条件没处理」记下来，下次重试时带着这些教训去改。

### 三种范式的对比总览

| 维度 | ReAct | Plan-and-Execute | Reflection |
|---|---|---|---|
| **核心思路** | 边想边干，走一步看一步 | 先规划再执行，全局把控 | 做完检查，不达标就改 |
| **解决的问题** | 单步灵活性 | 长任务跑偏 | 输出质量不够 |
| **独立流程？** | 是 | 是 | 否（叠加在前两者之上） |
| **灵活度** | 最高 | 较低（计划定了不好改） | 取决于底层范式 |
| **长任务稳定性** | 容易跑偏 | 结构清晰不跑偏 | 不改变任务完成能力 |
| **实现复杂度** | 最低 | 中等 | 在基础范式上叠加 |
| **Token 消耗** | 线性增长 | 可控 + 可用混合模型降本 | 基础上增加 30%-100% |
| **典型场景** | 信息搜索、客服、问答 | 报告生成、项目开发、调研 | 代码编写、法律文书、商业报告 |

### Token 消耗对比总结

以 5 步工具调用任务为基准（每步平均 2000 token）：

| 范式 | 总 Token 消耗 | 说明 |
|---|---|---|
| ReAct | ~30,000 | 输入逐轮递增，线性增长 |
| Plan-and-Execute | ~14,500 | 规划 1 次 + 执行 5 次 + 汇总 1 次 |
| Plan-and-Execute + 混合模型 | ~14,500（token）/ 成本降 70%+ | 执行阶段用便宜模型 |
| 任意范式 + Reflection | 基础上 +30%-100% | 每个反思点至少多 1 次 LLM 调用 |

## 常见追问与应对

**Q1: Reflection 为什么不是独立范式？面试官说它和 ReAct、Plan-and-Execute 并列，怎么反驳？**
> Reflection 没有自己的「任务完成流程」——它不定义 Agent 怎么从输入走到输出，只定义了「输出之后怎么检查和改进」。没有 ReAct 或 Plan-and-Execute 做底座，Reflection 无从附着。三者不在同一层面：ReAct 和 Plan-and-Execute 是「做事方式」，Reflection 是「质量保障」。面试官如果把它俩并列，是故意考察你是否理解这个层次区别。

**Q2: 实际项目中三种范式怎么组合？最常见的架构是什么？**
> 最常见的生产级架构是三层嵌套：外层用 Plan-and-Execute 定全局计划，每一步的执行内部用 ReAct 循环（因为单步执行也可能需要多轮工具调用），最后对整体输出做一次 Reflection 检查质量。在 LangGraph 这类框架里实现很自然。

**Q3: 什么时候该加动态 Replan？怎么判断？**
> 当执行过程中经常遇到「意外」时——比如外部 API 返回了非预期结果、数据源发生了变化、用户中途修改了需求。判断标准：如果跑评测集时发现 Plan-and-Execute 有超过 20% 的任务因为「计划不合理」而失败，就值得加动态 Replan。

**Q4: Reflexion 的「错题本」机制会不会越来越大，影响性能？**
> 会。经验教训如果无限制累积，上下文会越来越长，Token 消耗和延迟都会增加。实际做法：1）对教训做摘要压缩，每条控制在 1-2 句话；2）设置过期机制，长期没被用到的教训自动淘汰；3）用向量检索代替全文传入——只检索与当前任务相关的 Top-K 条教训。这也是 RAG 技术在 Agent 场景的应用。
