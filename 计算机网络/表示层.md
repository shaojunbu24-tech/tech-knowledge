# OSI 第六层：表示层（Presentation Layer）

> 分类：计算机网络
> 标签：表示层, 数据编码, TLS/SSL, 加密, 压缩, 序列化, 字符编码

## 面试精答

> 适合面试时口述的简洁版本（2-3分期能说完的要点）

表示层是 OSI 模型的"翻译官"——负责处理数据的**表示方式**，确保不同系统之间能互相理解对方的数据。

**三个核心功能**：
1. **数据格式转换**：不同系统的数据编码可能不同（大端序 vs 小端序、UTF-8 vs GBK），表示层负责统一格式
2. **加密与解密**：TLS/SSL 加密在这一层工作，保证数据在传输过程中不被窃听
3. **数据压缩与解压缩**：gzip 压缩 HTTP 响应、图片压缩等，减少传输数据量

**生活化类比**：两个说不同语言的人通过翻译交流——表示层就是翻译官，还负责把"机密内容"加密防止被偷听，把"啰嗦的话"压缩精简。

**重要认知**：和会话层一样，表示层在 TCP/IP 模型中被合并到了应用层。但它的功能无处不在——你访问 HTTPS 网站时的 TLS 加密、API 返回的 JSON 序列化、HTTP 请求的 gzip 压缩，都是表示层的工作。

## 原理深入详解

> 深入底层原理，理解"为什么"

### 一、为什么需要表示层

```
不同系统之间的"语言障碍"：

系统 A（x86, Linux, UTF-8）              系统 B（ARM, Windows, GBK）
  │                                        │
  │  "你好" → UTF-8 编码 → E4 BD A0 E5 A5 BD    │
  │  ────────────────────────────────→     │
  │         系统 B 用 GBK 解码 → "浣犲ソ"（乱码）│
  │                                        │

表示层的作用：
  系统 A：数据 → 表示层统一编码（如 UTF-8） → 发送
  系统 B：接收 → 表示层解码（从 UTF-8 转 GBK）→ 正确显示 "你好"
```

### 二、数据编码

#### 字符编码

计算机只认识 0 和 1，字符需要"编码"才能存储和传输。

```
字符编码的演进：

ASCII（1963年）
  - 7 位编码，128 个字符
  - 只覆盖英文字母、数字、标点
  - "A" = 65 = 01000001

Unicode（1991年）
  - 统一全球所有字符的编码标准
  - 目前收录超过 14 万个字符（中英日韩、emoji、古文字...）
  - "你" = U+4F60

UTF-8（Unicode 的实现方式之一）
  - 变长编码：ASCII 字符 1 字节，中文 3 字节，罕见字符 4 字节
  - 兼容 ASCII（英文内容完全不变）
  - 互联网上 98% 的网页使用 UTF-8
  - "A" = 41（1 字节）
  - "你" = E4 BD A0（3 字节）

GBK（中文编码）
  - 中文 2 字节，兼容 GB2312
  - 中文 Windows 系统默认编码
```

**为什么会有乱码**：发送方用 UTF-8 编码，接收方用 GBK 解码，编码方式不一致就会乱码。表示层的工作就是确保双方使用统一的编码。

#### 字节序（Endianness）

多字节数据在内存中怎么排列？

```
数值 0x12345678 在内存中的存储：

大端序（Big-Endian）：高位字节在低地址
  地址：  0x00  0x01  0x02  0x03
  数据：  0x12  0x34  0x56  0x78
  （人类阅读的顺序，网络字节序用大端）

小端序（Little-Endian）：低位字节在低地址
  地址：  0x00  0x01  0x02  0x03
  数据：  0x78  0x56  0x34  0x12
  （x86/ARM 处理器用小端，因为电路设计更高效）
```

**网络字节序**：TCP/IP 规定网络传输统一用大端序。发送前把主机字节序转成网络字节序（`htonl()`/`htons()`），接收后转回来（`ntohl()`/`ntohs()`）。这就是表示层的工作。

#### 序列化

把内存中的数据结构转成可以在网络上传输的字节流。

```
内存中的对象：         序列化后的 JSON：
User {                {
  name: "张三"          "name": "张三",
  age: 25               "age": 25,
  email: "a@b.com"      "email": "a@b.com"
}                      }

常见序列化格式：
  JSON    — 可读性好，Web API 主流
  XML     — 可读性好，配置文件/SOAP 协议
  Protobuf — 二进制，体积小速度快，gRPC 使用
  MessagePack — 二进制 JSON，比 JSON 紧凑
  Thrift  — Facebook 开源，跨语言 RPC
```

### 三、加密与解密

#### TLS/SSL（传输层安全）

HTTPS = HTTP + TLS。TLS 工作在表示层（在 TCP 之上、应用层之下）。

```
TLS 的工作流程（简化）：

1. 客户端发起连接，发送支持的加密套件列表
2. 服务端返回数字证书（包含公钥）和选定的加密套件
3. 客户端验证证书合法性（CA 签名验证）
4. 客户端生成随机对称密钥，用服务端公钥加密后发送
5. 服务端用私钥解密，拿到对称密钥
6. 双方用对称密钥加密后续通信

握手完成后：
  应用层数据 → TLS 加密 → TCP 传输 → TLS 解密 → 应用层
```

**对称加密 vs 非对称加密**（面试常问）：

| 维度 | 对称加密（AES） | 非对称加密（RSA） |
|---|---|---|
| 密钥数量 | 1 个（加密解密用同一个） | 2 个（公钥加密、私钥解密） |
| 速度 | 快（硬件加速） | 慢（数学运算复杂） |
| 安全性 | 密钥分发是难点 | 公钥可以公开，私钥自己保管 |
| TLS 中的角色 | 加密实际数据（快） | 加密对称密钥（安全分发） |

**TLS 结合了两者的优势**：用非对称加密安全地交换对称密钥，再用对称密钥加密数据（兼顾安全和性能）。

### 四、数据压缩

压缩的目的是减少传输数据量，节省带宽和传输时间。

```
HTTP 压缩流程：

1. 客户端请求：Accept-Encoding: gzip, deflate, br
   "我能接受这些压缩格式"

2. 服务端响应：Content-Encoding: gzip
   "我用 gzip 压缩了"
   （原始 10KB 的 HTML 压缩后只有 3KB）

3. 客户端收到后自动解压，渲染页面
```

**常见压缩算法**：

| 算法 | 特点 | 适用场景 |
|---|---|---|
| **gzip** | 压缩比好，通用 | HTTP 响应压缩、日志压缩 |
| **deflate** | gzip 的前身，略轻量 | HTTP 响应 |
| **Brotli（br）** | 比 gzip 压缩比高 15-25% | 现代 Web（Google 推广） |
| **LZ4** | 压缩/解压速度极快 | 实时数据传输、Kafka |
| **Snappy** | Google 开源，速度快 | Hadoop、Kafka |

## 常见追问与应对

**Q1: HTTPS 的 TLS 在哪一层工作？**
> TLS 工作在表示层（OSI 模型）或传输层之上（TCP/IP 模型）。它接收应用层的数据，加密后交给传输层（TCP）发送。这就是为什么 HTTPS 使用 443 端口而 HTTP 使用 80 端口——TLS 需要独立的连接建立过程。

**Q2: JSON 和 Protobuf 的序列化性能差多少？**
> 序列化后大小：Protobuf 约为 JSON 的 1/3~1/5（二进制格式不传字段名和花括号）；序列化速度：Protobuf 比 JSON 快 5-10 倍。代价是 Protobuf 不可读（二进制），且需要 `.proto` 文件定义数据结构。gRPC 默认用 Protobuf 就是追求高性能。

**Q3: 大端序和小端序是怎么来的？为什么没有统一？**
> 名字来自《格列佛游记》——两个国家因为鸡蛋应该从大头还是小头打开而开战。x86 用小端是因为电路设计更自然（最低有效位在最低地址），网络用大端是因为人类阅读习惯（高位在前）。两种都有道理所以没统一，表示层负责转换。
