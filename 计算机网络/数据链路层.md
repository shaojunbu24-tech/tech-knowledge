# OSI 第二层：数据链路层（Data Link Layer）

> 分类：计算机网络
> 标签：数据链路层, MAC 地址, 以太网帧, 交换机, ARP, CRC, CSMA/CD, VLAN

## 面试精答

> 适合面试时口述的简洁版本（2-3分期能说完的要点）

数据链路层负责在**相邻节点**之间提供可靠的数据帧传输。物理层只管传输比特流，不保证正确性；数据链路层在比特流的基础上加上了**帧结构、寻址（MAC 地址）、差错检测**。

**三个核心功能**：
1. **帧封装**：把网络层传下来的数据包封装成帧，加上帧头（源/目标 MAC 地址）和帧尾（CRC 校验）
2. **物理寻址**：通过 48 位 MAC 地址（如 `00:1A:2B:3C:4D:5E`）标识网络设备
3. **差错控制**：通过 CRC（循环冗余校验）检测传输过程中的比特错误

**关键设备**：交换机（Switch）。交换机根据 MAC 地址表**定向转发**帧，只发给目标设备，不广播。这和集线器（物理层设备，收到就广播）形成鲜明对比。

**ARP 协议**是数据链路层和网络层的桥梁：已知目标 IP，通过 ARP 广播请求获取目标 MAC 地址，再把 MAC 地址填入以太网帧头。

## 原理深入详解

> 深入底层原理，理解"为什么"

### 一、为什么需要数据链路层

物理层已经能把 0 和 1 从 A 送到 B 了，为什么还需要数据链路层？

```
物理层的问题：
1. 不知道 0 和 1 从哪开始、到哪结束（没有边界）
2. 不知道这些数据是给谁的（没有地址）
3. 不知道传输出错了没有（没有校验）
4. 发太快接收方可能处理不过来（没有流量控制）

数据链路层的解决方案：
1. 帧封装 — 定义数据的边界（帧头 + 数据 + 帧尾）
2. MAC 寻址 — 标识发送方和接收方
3. CRC 校验 — 检测传输错误
4. 流量控制 — 控制发送速率
```

### 二、MAC 地址

**什么是 MAC 地址**：Media Access Control Address，媒体访问控制地址。48 位（6 字节），通常用十六进制表示：`00:1A:2B:3C:4D:5E`。

```
MAC 地址结构（48 位）：

│←──── OUI（24位）────→│←─── 设备标识（24位）───→│
  00       1A       2B       3C       4D       5E

OUI（Organizationally Unique Identifier）：
  由 IEEE 分配给厂商，标识设备制造商
  如 00:1A:2B 是某厂商的 OUI

设备标识：
  厂商自行分配，保证同一 OUI 下不重复

第一位字节的最低位：
  0 = 单播地址（发给一个设备）
  1 = 组播地址（发给一组设备）

第一位字节的次低位：
  0 = 全局唯一（厂商烧录）
  1 = 本地管理（软件修改的）
```

**MAC 地址 vs IP 地址**（面试必问）：

| 维度 | MAC 地址 | IP 地址 |
|---|---|---|
| 长度 | 48 位（6 字节） | 32 位（IPv4）/ 128 位（IPv6） |
| 分配方式 | 网卡出厂烧录（理论上不变） | 网络配置分配（会变化） |
| 层次结构 | 扁平，没有层次 | 有层次（网络号 + 主机号） |
| 作用 | 标识"你是谁"（硬件身份） | 标识"你在哪里"（逻辑位置） |
| 类比 | 身份证号 | 邮寄地址 |

**为什么不能只用 MAC 地址路由**：MAC 地址是扁平的（没有"你在哪个网络"的信息），路由器无法根据 MAC 地址判断目标在哪个网络。IP 地址有网络号和主机号的层次结构，路由器可以逐级转发。

**MAC 地址真的不变吗**：不是。MAC 地址可以软件修改（`ifconfig eth0 hw ether 00:11:22:33:44:55`），手机连不同 WiFi 时可能用不同的 MAC 地址（MAC 随机化，隐私保护）。

### 三、以太网帧结构

以太网（Ethernet）是数据链路层最主流的协议，全球局域网几乎都在用以太网。

```
以太网帧（Ethernet II / DIX 帧）：

┌──────────┬──────────┬──────────┬──────────┬───────────────┬───────┐
│  前导码   │  帧起始符  │ 目标 MAC  │  源 MAC   │    类型       │ 数据   │  FCS  │
│  7 字节   │  1 字节   │  6 字节   │  6 字节   │   2 字节      │46-1500B│ 4 字节│
└──────────┴──────────┴──────────┴──────────┴───────────────┴───────┘
  ← 物理层加的 → │← ─ ─ ─ ─ ─ ─ ─ 数据链路层的帧 ─ ─ ─ ─ ─ ─ →│
                 │← 帧头（14 字节）→│          │← 帧尾（4 字节）→│

各字段详解：

前导码（Preamble）：7 字节，交替的 10101010...
  作用：让接收方的时钟同步到发送方的时钟频率
  这就是为什么前导码在"物理层加的"部分

帧起始符（SFD）：1 字节，10101011
  作用：标记帧的开始，最后两个 bit 是 11，告诉接收方"后面就是真正的帧了"

目标 MAC：6 字节，接收方的 MAC 地址
  如果是 FF:FF:FF:FF:FF:FF → 广播帧（发给同一网络所有设备）

源 MAC：6 字节，发送方的 MAC 地址

类型（EtherType）：2 字节，标识上层协议
  0x0800 = IPv4
  0x0806 = ARP
  0x86DD = IPv6
  0x0800 = IPv4

数据（Payload）：46 ~ 1500 字节
  最小 46 字节：以太网规定最小帧长 64 字节（用于冲突检测）
  64 - 14（帧头）- 4（FCS）= 46 字节
  如果数据不足 46 字节，填充到 46 字节
  最大 1500 字节：这就是 MTU（Maximum Transmission Unit）的由来

FCS（Frame Check Sequence）：4 字节，CRC-32 校验
```

**最小帧长 64 字节的原因**：

```
以太网使用 CSMA/CD 协议处理冲突：
  发送方在发送的同时检测冲突
  如果帧太短，发送方已经发完了才发现冲突 → 无法处理

64 字节的帧在 10 Mbps 以太网上传输时间 ≈ 51.2 μs
这个时间正好等于信号在最大长度（2500 米）的以太网上往返一次的时间
保证发送方在发完帧之前一定能检测到冲突
```

### 四、CRC 校验原理

**CRC（Cyclic Redundancy Check，循环冗余校验）**是数据链路层检测传输错误的核心手段。

**基本思路**：把数据当作一个大数，用一个"除数"（生成多项式）去除，得到的"余数"作为校验码附在数据后面。接收方收到后用同一个除数去除，如果余数为 0，说明数据没有错误。

```
简化例子（实际 CRC-32 要复杂得多）：

数据：101001
生成多项式：1101（CRC-3）

发送方计算：
  101001 ÷ 1101 = 商 ... 余数 001
  发送：101001 001（数据 + 余数）

接收方校验：
  101001001 ÷ 1101 = 商 ... 余数 000
  余数为 0 → 数据正确

如果传输中某个 bit 翻转：
  101101001 ÷ 1101 = 商 ... 余数 ≠ 0
  余数不为 0 → 检测到错误 → 丢弃该帧
```

**CRC 的能力**：
- 能检测所有单 bit 错误
- 能检测所有双 bit 错误
- 能检测所有奇数个 bit 错误
- 能检测所有长度 ≤ 32 的突发错误（CRC-32）

**注意**：CRC 只能**检测**错误，不能**纠正**错误。检测到错误后直接丢弃帧，不重传（重传由上层协议处理，如 TCP）。

### 五、交换机的工作原理

交换机是数据链路层最重要的设备。核心能力：**根据 MAC 地址定向转发帧**。

#### MAC 地址表（转发表）

交换机内部维护一张 MAC 地址表，记录"哪个 MAC 地址在哪个端口"：

```
交换机 MAC 地址表：

MAC 地址            端口     类型      老化时间
00:11:22:33:44:55   Port 1   动态学习   300s
00:AA:BB:CC:DD:EE   Port 3   动态学习   300s
FF:FF:FF:FF:FF:FF   ALL      广播       —
```

#### 学习过程

```
初始状态：MAC 地址表为空

1. 设备 A（MAC: AA）从 Port 1 发送帧到设备 B（MAC: BB）
   交换机记录：AA → Port 1（从帧的源 MAC 学到的）
   目标 MAC: BB 不在表中 → 广播到所有端口（除 Port 1）

2. 设备 B（MAC: BB）从 Port 3 收到帧后回复
   交换机记录：BB → Port 3
   目标 MAC: AA 在表中 → 只转发到 Port 1（定向转发）

3. 后续 A 和 B 通信，交换机直接定向转发，不再广播
```

#### 转发决策流程

```
交换机收到一个帧：

1. 学习：记录源 MAC → 入端口的映射
2. 查表：查找目标 MAC 在哪个端口
   ├── 找到了 → 转发到对应端口（单播）
   │            如果对应端口就是入端口 → 丢弃（过滤）
   └── 没找到 → 广播到所有端口（除入端口）
```

#### 交换机 vs 集线器

| 维度 | 集线器（Hub） | 交换机（Switch） |
|---|---|---|
| 工作层级 | 物理层 | 数据链路层 |
| 寻址能力 | 无（不认识 MAC 地址） | 有（根据 MAC 转发） |
| 转发方式 | 广播到所有端口 | 定向转发到目标端口 |
| 带宽 | 所有端口共享 | 每个端口独享 |
| 冲突域 | 整个 Hub 是一个冲突域 | 每个端口是独立的冲突域 |
| 安全性 | 所有设备都能收到数据 | 只有目标设备收到 |

### 六、ARP 协议

**ARP（Address Resolution Protocol，地址解析协议）**解决一个核心问题：已知目标 IP 地址，怎么知道目标 MAC 地址？

**为什么需要 ARP**：网络层用 IP 地址，数据链路层用 MAC 地址。发送数据时帧头需要填目标 MAC 地址，但上层只给了 IP 地址。ARP 就是这个"翻译器"。

```
ARP 工作流程：

主机 A（IP: 192.168.1.10, MAC: AA）要发数据给主机 B（IP: 192.168.1.20）

1. A 先查本地 ARP 缓存表
   arp -a → 192.168.1.20 → BB → 找到了！直接用

2. 如果缓存中没有：
   A 广播 ARP 请求（目标 MAC = FF:FF:FF:FF:FF:FF）
   "谁是 192.168.1.20？请告诉我你的 MAC 地址"

3. 网络中所有设备收到广播
   - 设备 C（IP: 192.168.1.30）：不是我的 IP，忽略
   - 设备 B（IP: 192.168.1.20, MAC: BB）：是我的！

4. B 单播 ARP 应答给 A
   "我是 192.168.1.20，我的 MAC 是 BB"

5. A 把 BB 记入 ARP 缓存表，后续直接使用
   ARP 缓存通常有老化时间（默认 20-30 分钟）
```

**ARP 只在同一网络内工作**：如果目标 IP 不在同一网段，不需要知道目标 MAC，而是把帧发给默认网关（路由器）。路由器的 MAC 地址通过 ARP 获取。

### 七、VLAN（虚拟局域网）

**为什么需要 VLAN**：一个交换机下连了财务部、研发部、行政部的电脑，所有人在同一个广播域——ARP 广播、DHCP 广播所有人都收到。既浪费带宽又不安全。

**VLAN 的作用**：在逻辑上把一个物理交换机划分成多个**独立的广播域**。

```
物理连接：

  PC1(财务) ─── Port 1 ─┐
  PC2(研发) ─── Port 2 ─┤
  PC3(财务) ─── Port 3 ─┼── 交换机
  PC4(研发) ─── Port 4 ─┤
  PC5(行政) ─── Port 5 ─┘

配置 VLAN 后（逻辑隔离）：

  VLAN 10（财务）：Port 1, Port 3  ← PC1 和 PC3 互相通信
  VLAN 20（研发）：Port 2, Port 4  ← PC2 和 PC4 互相通信
  VLAN 30（行政）：Port 5          ← PC5 独立

  不同 VLAN 之间不能直接通信（除非通过路由器/三层交换机）
```

**VLAN 的实现**：在以太网帧头中插入一个 4 字节的 802.1Q 标签，包含 VLAN ID（12 位，支持 4096 个 VLAN）。

## 常见追问与应对

**Q1: 交换机的 MAC 地址表能存多少条？满了怎么办？**
> 通常能存几千到几万条（企业级交换机）。满了以后新学到的 MAC 地址对应的帧会被广播（退化为 Hub 的行为）。MAC 地址表有老化机制（默认 5 分钟），长期不通信的条目会被清除。

**Q2: ARP 欺骗（ARP Spoofing）是什么？怎么防御？**
> 攻击者伪造 ARP 应答，告诉受害者"网关的 MAC 是我的 MAC"，受害者就把所有流量发给攻击者（中间人攻击）。防御：1）静态 ARP 绑定（`arp -s` 绑定 IP 和 MAC）；2）交换机上启用动态 ARP 检测（DAI）；3）使用 HTTPS 加密流量（即使被中间人也无法解密）。

**Q3: 为什么以太网帧最大 1500 字节？能改大吗？**
> 1500 字节是 1980 年代以太网标准定的，是性能和延迟的折中——帧太长会占用信道导致其他设备等待太久，帧太短则协议头占比太大浪费带宽。可以改大：巨帧（Jumbo Frame）支持 9000 字节，在数据中心内部使用可以提升吞吐量，但互联网上不支持（沿途设备可能不认）。
