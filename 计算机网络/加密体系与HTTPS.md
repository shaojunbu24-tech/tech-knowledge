# 对称加密、非对称加密与 HTTPS 中的应用

> 分类：计算机网络
> 标签：加密, 对称加密, 非对称加密, AES, RSA, HTTPS, TLS, 数字证书, CA, 哈希

## 面试精答

> 适合面试时口述的简洁版本（2-3分钟能说完的要点）

### 先理解"加密"解决什么问题

网络传输就像寄明信片——任何人都能看到上面的内容。加密就是把明信片装进一个**只有收件人能打开的密封信封**里。

网络安全面临三类威胁：
- **窃听**：别人偷看内容 → 加密解决
- **篡改**：别人改了内容 → 完整性校验解决
- **冒充**：别人假装是服务器 → 身份认证解决

HTTPS 用一套完整的加密体系同时解决了这三个问题。

### 对称加密（Symmetric Encryption）

**核心**：加密和解密用**同一把密钥**。

```
发送方：明文 + 密钥 K → 加密算法 → 密文
接收方：密文 + 密钥 K → 解密算法 → 明文

就像一把锁配一把钥匙：
  加密 = 用钥匙锁上保险箱
  解密 = 用同一把钥匙打开保险箱
```

**优点**：速度快（AES 加密可达 GB/s 级别），适合加密大量数据。

**缺点**：密钥分发问题——你怎么安全地把密钥告诉对方？如果密钥在传输中被截获，加密就没意义了。

**常见算法**：
- **AES（Advanced Encryption Standard）**：当前最主流，密钥长度 128/192/256 位
- **DES**：已淘汰，密钥只有 56 位，可被暴力破解
- **ChaCha20**：Google 推广，在移动端比 AES 更快

### 非对称加密（Asymmetric Encryption）

**核心**：一对密钥——**公钥（Public Key）**和**私钥（Private Key）**。公钥加密的数据只有私钥能解，私钥签名的数据公钥能验证。

```
场景一：加密通信
  发送方：明文 + 对方公钥 → 加密 → 密文
  接收方：密文 + 自己的私钥 → 解密 → 明文

  公钥是公开的（任何人都能拿到）
  私钥只有自己持有
  → 任何人都能给你发加密消息，只有你能解密

场景二：数字签名（反过来用）
  发送方：消息 + 自己的私钥 → 签名
  接收方：签名 + 对方公钥 → 验证

  → 证明"这消息确实是我发的，没被篡改"
```

**优点**：解决了密钥分发问题——公钥可以公开，不需要安全地传递。

**缺点**：速度极慢（比对称加密慢 100-1000 倍），不适合加密大量数据。

**常见算法**：
- **RSA**：最经典，密钥长度 2048/4096 位
- **ECC（Elliptic Curve Cryptography）**：更短的密钥达到同等安全强度，移动端友好
- **Ed25519**：基于椭圆曲线，现代应用推荐

### 两种加密的对比

| 维度 | 对称加密 | 非对称加密 |
|---|---|---|
| 密钥数量 | 1 个（双方共享） | 2 个（公钥 + 私钥） |
| 速度 | 极快（AES ~GB/s） | 很慢（RSA ~KB/s） |
| 安全性 | 密钥泄露就全完了 | 私钥泄露才出问题 |
| 核心问题 | 密钥怎么安全给对方？ | 公钥怎么证明是真的？ |
| 适合场景 | 加密大量数据 | 密钥交换、身份认证 |

### HTTPS 怎么把两者结合起来

**核心思路**：用非对称加密安全地交换对称密钥，再用对称密钥加密实际数据。**各取所长**。

```
HTTPS（TLS 握手）完整流程：

第 1 步：TCP 三次握手（建立连接）
  客户端 ──SYN──→ 服务器
  客户端 ←─SYN+ACK── 服务器
  客户端 ──ACK──→ 服务器

第 2 步：TLS 握手（协商加密）
  ┌───────────────────────────────────────────────┐
  │ 客户端                                         │ 服务器
  │                                                │
  │ ──Client Hello──→                              │
  │   "我支持这些加密套件"                           │
  │                                                │
  │ ←─Server Hello + 证书──                        │
  │   "我们用这个套件，这是我的证书"                   │
  │   证书里包含：服务器公钥 + CA 签名               │
  │                                                │
  │ 客户端验证证书：                                 │
  │   1. 证书是否被信任的 CA 签发？                   │
  │   2. 证书有没有过期？                            │
  │   3. 证书的域名和访问的域名一致吗？               │
  │                                                │
  │ 客户端生成一个随机数（Pre-Master Secret）         │
  │   用服务器公钥加密后发送：                        │
  │ ──加密的 Pre-Master Secret──→                   │
  │   （只有服务器的私钥能解密）                      │
  │                                                │
  │ 双方根据 Pre-Master Secret + 两个随机数          │
  │   计算出同一个对称密钥（Session Key）             │
  │                                                │
  │ ──Finished（用 Session Key 加密）──→            │
  │ ←─Finished（用 Session Key 加密）──             │
  └───────────────────────────────────────────────┘

第 3 步：HTTP 数据传输
  所有 HTTP 请求和响应都用 Session Key（对称加密）传输
  → 快！因为是对称加密
  → 安全！因为 Session Key 是用非对称加密安全交换的
```

**为什么不全用非对称加密**：太慢了。一个网页几百 KB，用 RSA 加密要几百毫秒，用 AES 只要几毫秒。TLS 握手只占整个通信的一小部分，绝大部分数据传输用对称加密。

### 数字证书与 CA

**核心问题**：非对称加密的公钥是公开的，但你怎么知道这个公钥真的是"那个服务器"的，而不是中间人伪造的？

```
没有证书的世界：
  客户端 → "请给我你的公钥" → 中间人截获
  中间人 → 把自己的公钥给客户端
  客户端 → 用中间人的公钥加密数据
  中间人 → 用自己的私钥解密（窃听成功！）

有证书的世界：
  服务器 → 向 CA 申请证书
  CA（权威机构）→ 验证服务器身份后，用自己的私钥给证书签名
  证书内容：服务器公钥 + 服务器信息 + CA 签名

  客户端收到证书后：
  1. 浏览器内置了主流 CA 的公钥（Root CA）
  2. 用 CA 的公钥验证证书上的签名
  3. 签名验证通过 → 证书是真的 → 证书里的公钥可信

  中间人无法伪造证书：
    他没有 CA 的私钥 → 无法伪造 CA 签名 → 浏览器验证失败 → 警告
```

**证书信任链**：

```
Root CA（浏览器内置，绝对信任）
  │ 签发
  ↓
Intermediate CA（中间证书机构）
  │ 签发
  ↓
Server Certificate（服务器证书）
  包含：域名、公钥、有效期、CA 签名

验证过程：
  浏览器用 Root CA 公钥验证 Intermediate CA 证书
  → 再用 Intermediate CA 公钥验证 Server 证书
  → 整条链都通过才信任

为什么要有中间 CA：
  Root CA 的私钥极度重要（泄露后果灾难性）
  日常签发证书的工作交给 Intermediate CA
  Root CA 的私钥尽量少用，离线保存
```

### 哈希与消息认证码（MAC）

加密保证"别人看不懂"，但怎么保证"没被篡改"？

```
哈希（Hash）：
  任意长度的数据 → 哈希算法 → 固定长度的摘要（指纹）
  SHA-256：任意数据 → 256 位的摘要

  特点：
  1. 不可逆（不能从摘要反推出原数据）
  2. 雪崩效应（改一个 bit，摘要完全不同）
  3. 抗碰撞（几乎不可能找到两个不同的数据产生相同摘要）

TLS 中的 MAC（Message Authentication Code）：
  发送方：数据 + Session Key → HMAC 算法 → MAC 值
  发送：密文 + MAC 值

  接收方：解密得到数据 + Session Key → HMAC 算法 → MAC 值
  对比收到的 MAC 值和自己算的 MAC 值
  一致 → 数据没被篡改
  不一致 → 数据被篡改，丢弃
```

## 常见追问与应对

**Q1: HTTPS 能被破解吗？中间人攻击怎么防御？**
> 理论上，如果攻击者能拿到 CA 的私钥或者往浏览器里植入自己的 Root CA，就能伪造证书。但实际中极难做到。防御：1）证书透明度（CT）日志——所有签发的证书都记录在公开日志中，可审计；2）HPKP（已废弃）/Certificate Transparency 防止恶意签发；3）用户不要随意安装不明来源的 Root CA 证书。

**Q2: TLS 1.2 和 TLS 1.3 有什么区别？**
> TLS 1.3 做了三个大改进：1）握手从 2-RTT 缩短到 1-RTT（首次）和 0-RTT（重连），更快；2）去掉了不安全的加密套件（DES、3DES、RC4、RSA 密钥交换全部禁用），只保留 5 个强密码套件；3）前向保密（Forward Secrecy）成为强制——即使服务器私钥泄露，历史通信也无法被解密（因为每次会话的密钥都是临时生成的）。

**Q3: 既然 HTTPS 这么安全，为什么不是所有网站都用？**
> 现在基本都用了。Let's Encrypt 提供免费证书，浏览器对 HTTP 网站标"不安全"，主流网站已全面 HTTPS。早期不用是因为：1）证书要钱；2）加密有性能开销（现在 TLS 1.3 几乎无感）；3）CDN/代理缓存 HTTPS 内容更复杂。

**Q4: RSA 和 ECC 有什么区别？实际中用哪个？**
> RSA 基于大数分解难题，ECC 基于椭圆曲线离散对数难题。ECC 用更短的密钥达到同等安全强度：RSA 2048 位 ≈ ECC 256 位。ECC 更快、更省资源，适合移动端。现代 HTTPS 证书同时包含 RSA 和 ECC 密钥（双证书），服务器根据客户端支持情况选择。
