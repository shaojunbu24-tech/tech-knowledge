# 进程、线程、协程分别是什么，区别与联系

> 分类：操作系统
> 标签：进程, 线程, 协程, 并发, 并行, 上下文切换, 调度

## 面试精答

> 适合面试时口述的简洁版本（2-3分钟能说完的要点）

**进程**是操作系统资源分配的最小单位，每个进程拥有独立的内存空间（代码段、数据段、堆、栈）。进程之间互相隔离，一个进程崩溃不会影响其他进程。但进程的创建和切换开销大（需要切换页表、刷新 TLB 等）。

**线程**是 CPU 调度的最小单位，同一进程下的多个线程共享进程的内存空间和资源（代码段、数据段、堆），但每个线程有自己独立的栈和寄存器。线程切换比进程轻量（不需要切换地址空间），但线程间共享内存带来了同步问题（竞态条件、死锁）。

**协程**是用户态的轻量级线程，由程序员（或运行时库）控制调度，而非操作系统内核。协程在自己的栈上保存/恢复上下文，切换时不需要内核介入，开销极小（纳秒级 vs 微秒级）。一个线程内可以运行成千上万个协程。Go 的 goroutine、Python 的 async/await、Kotlin 的 suspend 函数都是协程的实现。

**核心区别一句话**：
- 进程：资源隔离，安全但重
- 线程：资源共享，轻量但需要同步
- 协程：用户态调度，极轻但不能利用多核（需要配合多线程）

**联系**：三者是层层包含的关系——一个进程可以包含多个线程，一个线程可以包含多个协程。协程不能脱离线程存在，它最终还是要跑在某个线程上。

## 原理深入详解

> 深入底层原理，理解"为什么"

### 一、进程（Process）

**本质**：进程是程序的一次执行实例。操作系统通过 PCB（Process Control Block，进程控制块）来管理进程，PCB 中保存了进程 ID、状态、优先级、程序计数器、内存指针、上下文数据等信息。

**进程的内存布局**：

```
高地址
┌──────────────┐
│   栈（Stack）  │  ← 函数调用、局部变量（向下增长）
├──────────────┤
│              │
│   空闲区域    │
│              │
├──────────────┤
│   堆（Heap）   │  ← malloc/new 动态分配（向上增长）
├──────────────┤
│  数据段（Data） │  ← 全局变量、静态变量
├──────────────┤
│  代码段（Text） │  ← 编译后的机器指令（只读）
└──────────────┘
低地址
```

**进程的状态转换**：

```
                    调度
     ┌──────┐  ────────→  ┌──────┐
     │ 就绪  │             │ 运行  │
     │ Ready │  ←────────  │Running│
     └──────┘   时间片用完  └──┬───┘
       ↑  ↑                     │
       │  │    I/O完成           │ 等待I/O或事件
       │  └─────────┐           ↓
       │       ┌────┴────┐
       │       │  阻塞    │
       │       │ Blocked  │
       │       └─────────┘
       │ 创建
     ┌─┴──────┐
     │  新建   │
     │  New    │
     └────────┘
```

**进程间通信（IPC）**：由于进程内存隔离，必须通过 IPC 机制通信：

| 方式 | 原理 | 特点 |
|---|---|---|
| 管道（Pipe） | 半双工字节流，父子进程间使用 | 简单，但只限父子进程 |
| 命名管道（FIFO） | 文件系统中的特殊文件 | 无亲缘关系进程也可通信 |
| 消息队列 | 内核中的消息链表 | 有格式，可按类型读取 |
| 共享内存 | 多个进程映射同一块物理内存 | 最快，但需要同步机制 |
| 信号量（Semaphore） | 计数器，控制共享资源访问 | 常和共享内存配合使用 |
| 信号（Signal） | 软件中断，通知进程某事件发生 | 只能传递信号编号，信息量少 |
| Socket | 网络通信接口 | 可跨机器通信 |

**进程的上下文切换开销**：
1. 保存当前进程的 CPU 寄存器到 PCB
2. 切换虚拟地址空间（切换页表基址寄存器，刷新 TLB）
3. 刷新 CPU 缓存（L1/L2/L3 Cache）
4. 恢复目标进程的 CPU 寄存器
5. 跳转到目标进程上次暂停的位置继续执行

其中第 2、3 步是开销最大的部分，TLB 和 Cache 的失效会导致后续大量内存访问变慢。

### 二、线程（Thread）

**本质**：线程是进程内的执行流。操作系统通过 TCB（Thread Control Block）管理线程。

**线程共享与独占的资源**：

```
进程内的多个线程：

    线程1        线程2        线程3
   ┌──────┐    ┌──────┐    ┌──────┐
   │ 独立栈 │    │ 独立栈 │    │ 独立栈 │    ← 各自独立
   │ 寄存器 │    │ 寄存器 │    │ 寄存器 │    ← 各自独立
   │ 程序   │    │ 程序   │    │ 程序   │
   │ 计数器 │    │ 计数器 │    │ 计数器 │    ← 各自独立
   └──┬───┘    └──┬───┘    └──┬───┘
      │           │           │
      └───────────┼───────────┘
                  ↓
         ┌──────────────┐
         │  共享内存空间  │    ← 代码段、数据段、堆
         │  共享文件描述符│
         │  共享信号处理  │
         └──────────────┘
```

**为什么线程共享内存**：因为同一进程的线程在同一个虚拟地址空间中运行，它们看到的是同一份代码段、数据段和堆。每个线程只是有自己的栈和寄存器状态。共享内存使得线程间通信非常高效（直接读写共享变量），但也带来了竞态条件问题。

**线程的实现模型**：

| 模型 | 说明 | 例子 |
|---|---|---|
| 1:1（内核级线程） | 每个用户线程对应一个内核线程 | Linux（pthread）、Windows |
| N:1（用户级线程） | 多个用户线程映射到一个内核线程 | 早期 Java 绿色线程 |
| M:N（混合模型） | M 个用户线程映射到 N 个内核线程 | Go（goroutine → 系统线程）、Rust tokio |

Linux 的线程实现比较特殊——在内核层面，线程和进程用的是同一个数据结构（`task_struct`），线程本质上是"共享地址空间的进程"。Linux 通过 `clone()` 系统调用创建线程，通过参数控制哪些资源共享、哪些独立。

**线程同步问题**：

共享内存带来的经典问题——竞态条件（Race Condition）：

```
两个线程同时执行 counter++（counter 初始为 0）：

理想情况：        竞态条件：
线程A: 读 counter=0    线程A: 读 counter=0
线程A: counter=1       线程B: 读 counter=0  ← 还没来得及写回
线程A: 写回 1          线程A: 写回 1
线程B: 读 counter=1    线程B: counter=1
线程B: counter=2       线程B: 写回 1  ← 覆盖了线程A的结果！
线程B: 写回 2

结果: 2（正确）        结果: 1（错误！）
```

**线程同步机制**：
- **互斥锁（Mutex）**：保证同一时刻只有一个线程进入临界区
- **读写锁（RWLock）**：读可并发，写独占
- **条件变量（Condition Variable）**：线程间通知等待机制
- **自旋锁（Spinlock）**：忙等待，适合临界区很短的场景
- **信号量（Semaphore）**：控制同时访问资源的线程数量

**线程 vs 进程的上下文切换开销**：
线程切换不需要切换地址空间（页表、TLB），只需要保存/恢复寄存器和栈指针。实测中，线程切换大约是进程切换的 **1/5 ~ 1/10** 开销。

### 三、协程（Coroutine）

**本质**：协程是比线程更轻量的执行单元。关键区别在于**调度器在哪**——进程和线程由操作系统内核调度（抢占式），协程由用户态的运行时库调度（协作式）。

**协程的执行模型**：

```
一个线程内的多个协程：

  线程（内核调度）
  ┌──────────────────────────────────────────┐
  │                                          │
  │  协程A         协程B         协程C        │
  │ ┌─────┐      ┌─────┐      ┌─────┐      │
  │ │ 运行 │ ──→  │ 挂起 │      │ 就绪 │      │
  │ └──┬──┘      └─────┘      └─────┘      │
  │    │ yield/await              ↑          │
  │    ↓                          │          │
  │ ┌─────┐      ┌─────┐         │          │
  │ │ 挂起 │      │ 运行 │ ───────┘          │
  │ └─────┘      └──┬──┘                    │
  │                 │ yield/await             │
  │                 ↓                         │
  │              ┌─────┐                     │
  │              │ 挂起 │                     │
  │              └─────┘                     │
  └──────────────────────────────────────────┘

  调度由用户态运行时完成，内核无感知
```

**协程切换为什么这么快**：
1. 不需要陷入内核态（没有系统调用）
2. 不需要切换地址空间（页表、TLB 不变）
3. 只需要保存/恢复少量寄存器和栈指针
4. CPU Cache 完全不受影响

开销对比：进程切换 ~几微秒，线程切换 ~1-2 微秒，协程切换 ~几十纳秒。

**协程的两种风格**：

| 风格 | 特点 | 代表 |
|---|---|---|
| **非对称协程（Asymmetric）** | 有调用者和被调用者的区分，类似函数调用（yield 返回给调用者） | Python（async/await）、Lua |
| **对称协程（Symmetric）** | 没有主从关系，协程之间可以直接切换控制权 | Go（goroutine） |

**语言级实现对比**：

| 语言 | 协程实现 | 调度方式 | 栈大小 |
|---|---|---|---|
| Go | goroutine | M:N 调度（runtime 调度器），协作+抢占 | 初始 2KB，可动态增长 |
| Python | async/await | 事件循环（asyncio），单线程 | 固定，与线程共享 |
| Java (Virtual Thread) | 虚拟线程 | M:N 调度（ForkJoinPool） | 存储在堆上，动态 |
| Kotlin | suspend 函数 | 协程调度器（CoroutineDispatcher） | 编译器状态机实现 |
| Rust | async/await + tokio | 事件循环 + 工作窃取 | 编译器状态机实现 |

**Go goroutine 的调度模型（GMP 模型）**：

```
G (Goroutine)    — 协程
M (Machine)      — 操作系统线程
P (Processor)    — 逻辑处理器（调度的上下文）

每个 P 有一个本地运行队列，存放待运行的 G
M 必须绑定一个 P 才能执行 G
当某个 P 的队列空了，会从其他 P "偷" G（work stealing）

    P1          P2          P3
  ┌─────┐    ┌─────┐    ┌─────┐
  │ G G  │    │ G G  │    │ G G  │
  │ G G  │    │  ↑   │    │ G    │
  └──┬───┘    └──┼───┘    └──┬───┘
     │           │偷          │
  ┌──▼───┐    ┌──▼───┐    ┌──▼───┐
  │  M1  │    │  M2  │    │  M3  │   ← 操作系统线程
  └──────┘    └──────┘    └──────┘
```

**为什么 goroutine 可以开几十万个**：
1. 初始栈只有 2KB（线程通常 1-8MB），内存占用极小
2. 由 Go runtime 调度，不需要内核参与
3. M:N 模型，几十万个 goroutine 只需要几十个系统线程

### 四、三者的核心对比

| 维度 | 进程 | 线程 | 协程 |
|---|---|---|---|
| **定义** | 资源分配的最小单位 | CPU 调度的最小单位 | 用户态的轻量级线程 |
| **调度者** | 操作系统内核 | 操作系统内核 | 用户态运行时库 |
| **调度方式** | 抢占式 | 抢占式 | 协作式（主动 yield） |
| **上下文切换开销** | 大（~几 μs） | 中（~1-2 μs） | 小（~几十 ns） |
| **内存隔离** | 完全隔离 | 共享地址空间 | 共享线程的地址空间 |
| **创建开销** | 很大（fork） | 中（pthread_create） | 极小（go func） |
| **通信方式** | IPC（管道、共享内存等） | 共享变量（需同步） | 直接通信（共享变量） |
| **多核利用** | 天然支持 | 天然支持 | 需配合多线程 |
| **稳定性** | 高（隔离性好） | 中（一个线程崩可能拖垮整个进程） | 取决于线程 |
| **数量级** | 几十~几百 | 几百~几千 | 几万~几十万 |
| **典型使用** | Chrome 多进程架构、Nginx worker | Java 多线程、C++ std::thread | Go goroutine、Python asyncio |

### 五、三者如何配合使用

实际系统中，三者不是互斥的，而是**层层嵌套**：

```
Nginx 的架构：
  Master 进程
    ├── Worker 进程 1
    │     └── 线程池（处理磁盘 I/O）
    ├── Worker 进程 2
    │     └── 线程池
    └── Worker 进程 N

Go HTTP 服务器：
  进程
    └── 多个操作系统线程（由 Go runtime 管理）
          └── 几十万 goroutine（M:N 调度）

现代微服务架构：
  多个进程（容器/服务）
    └── 每个进程内多个线程
          └── 每个线程内多个协程
```

**选型建议**：
- 需要隔离性（安全、容错）→ 多进程（如 Chrome 每个标签页一个进程）
- 需要利用多核 + 共享数据 → 多线程
- 高并发 I/O 密集型 → 协程（如 Web 服务器处理海量连接）
- CPU 密集型计算 → 多进程/多线程（协程对 CPU 密集型帮助不大）

## 常见追问与应对

**Q1: 协程能利用多核吗？**
> 协程本身不能。协程运行在用户态，操作系统看不到协程，一个线程内的所有协程只能跑在一个 CPU 核心上。要利用多核，必须配合多线程——Go 的 GMP 模型就是这样做的：多个系统线程（M）分别绑定到不同 CPU 核心，每个线程上运行不同的 goroutine，从而实现多核并行。

**Q2: 线程和进程的选择怎么考虑？实际中怎么用的？**
> 需要隔离性用多进程（如 Nginx worker 进程互相隔离、Chrome 多进程架构），需要高性能共享数据用多线程（如数据库内部并发）。现代实践中，两者经常结合使用——Nginx 就是用多进程 + 每个进程内的线程池。Java 通常是单进程多线程，配合线程池管理。

**Q3: 协程的"协作式调度"有什么缺点？**
> 协作式调度的核心问题是**一个协程不主动 yield，其他协程就得不到执行机会**。如果某个协程执行了 CPU 密集型计算不释放控制权，整个线程会被阻塞。Go 在 1.14 版本引入了基于信号的抢占式调度来解决这个问题——即使 goroutine 不主动 yield，runtime 也会在特定安全点强制抢占。

**Q4: 线程池是什么？为什么要用线程池？**
> 线程池是预先创建一组线程复用，避免频繁创建销毁线程的开销。核心参数：核心线程数、最大线程数、任务队列、拒绝策略。线程池解决了两个问题：1）控制并发数，防止系统过载；2）复用线程，减少创建销毁的开销。Java 的 ThreadPoolExecutor、Go 的 goroutine + channel 都是对线程池思想的实现。
