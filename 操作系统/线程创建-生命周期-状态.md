# 线程的创建方式、生命周期与状态转换

> 分类：操作系统
> 标签：线程, 线程创建, 线程生命周期, 线程状态, Java 线程, 操作系统调度

## 面试精答

> 适合面试时口述的简洁版本（2-3分钟能说完的要点）

### 线程的创建

**操作系统层面**：线程通过系统调用创建。Linux 用 `clone()` 系统调用（flags 参数控制资源共享程度），Windows 用 `CreateThread()` API。创建线程时，操作系统需要分配 TCB、独立的栈空间、设置寄存器初始值，然后加入调度队列。

**Java 层面**（面试高频）：有四种方式，本质都是启动 Thread 对象：

| 方式 | 核心 | 特点 |
|---|---|---|
| 继承 Thread 类 | 重写 run() 方法 | 简单，但 Java 单继承限制了扩展性 |
| 实现 Runnable 接口 | 传入 Thread 构造函数 | 推荐方式，解耦任务与线程 |
| 实现 Callable 接口 | 配合 FutureTask 可获取返回值 | 需要返回结果时使用 |
| 线程池（ExecutorService） | 提交 Runnable/Callable | 生产环境首选，复用线程、控制并发数 |

前三种方式底层都是调用 `Thread.start()` → JVM 调用本地方法 `start0()` → 操作系统创建线程。直接调用 `run()` 不会创建新线程，只是在当前线程同步执行。

### 线程的生命周期

**操作系统的五状态模型**：

```
  新建 ──→ 就绪 ─⇄── 运行 ──→ 终止
              ↑       │
              │    等待事件
              │       ↓
              └── 阻塞
```

- **新建（New）**：线程对象已创建，尚未启动
- **就绪（Ready）**：已加入调度队列，等待 CPU 分配时间片
- **运行（Running）**：正在 CPU 上执行
- **阻塞（Blocked）**：等待某个事件（I/O 完成、获取锁、等待通知）
- **终止（Terminated）**：执行完毕或异常退出

**Java 线程的六状态**（面试重点，`Thread.State` 枚举）：

```
NEW ──start()──→ RUNNABLE ⇄ RUNNING（JVM 不区分，统一叫 RUNNABLE）
                     │
              ├─ wait() ─────────→ WAITING
              ├─ join() ─────────→ WAITING
              ├─ LockSupport.park() → WAITING
              │
              ├─ sleep(ms) ───────→ TIMED_WAITING
              ├─ wait(ms) ────────→ TIMED_WAITING
              ├─ LockSupport.parkNanos() → TIMED_WAITING
              │
              ├─ 等待 synchronized 锁 → BLOCKED
              │
              └─ run() 执行完毕 ──→ TERMINATED
```

核心区别：Java 把操作系统的"就绪"和"运行"合并为 `RUNNABLE`，把"阻塞"细分为 `BLOCKED`（等锁）、`WAITING`（无限期等待）、`TIMED_WAITING`（限时等待）三种。

## 原理深入详解

> 深入底层原理，理解"为什么"

### 一、操作系统层面如何创建线程

**Linux 的线程创建**：

```c
// pthread_create 的底层实现
int pthread_create(pthread_t *tid, const pthread_attr_t *attr,
                   void *(*start_routine)(void*), void *arg) {
    // clone() 系统调用是关键
    // flags 参数控制哪些资源共享
    clone(start_routine,      // 线程入口函数
          stack_addr,          // 新线程的栈地址
          CLONE_VM |           // 共享虚拟地址空间
          CLONE_FS |           // 共享文件系统信息
          CLONE_FILES |        // 共享文件描述符表
          CLONE_SIGHAND |      // 共享信号处理器
          CLONE_THREAD,        // 放入同一线程组
          arg);                // 传递给线程的参数
}
```

**clone() vs fork() 的本质区别**：

| 系统调用 | 共享内容 | 独立内容 | 用途 |
|---|---|---|---|
| `fork()` | 几乎不共享（写时复制） | 完整的地址空间副本 | 创建进程 |
| `clone(CLONE_VM\|CLONE_THREAD)` | 地址空间、文件描述符、信号处理 | 栈、寄存器、线程 ID | 创建线程 |

Linux 中线程和进程的内核数据结构是同一个（`task_struct`），区别仅在于创建时 `clone()` 的 flags 参数不同——共享的资源多就是"线程"，共享的资源少就是"进程"。

**创建线程时的开销**：

1. 分配 TCB（线程控制块）~ 几 KB
2. 分配栈空间（默认 1-8 MB，取决于系统）
3. 初始化寄存器（设置栈指针、程序计数器指向线程入口函数）
4. 将线程加入调度器的运行队列
5. 触发调度器重新评估（可能触发立即切换）

这就是为什么生产环境用**线程池**——预先创建好线程，避免频繁创建销毁的开销。

### 二、Java 的四种创建方式详解

**方式一：继承 Thread 类**

```java
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("线程运行: " + getName());
    }
}

new MyThread().start();
```

局限：Java 单继承，继承了 Thread 就不能再继承其他类。任务代码和线程机制耦合在一起。

**方式二：实现 Runnable 接口（推荐）**

```java
class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("任务运行: " + Thread.currentThread().getName());
    }
}

new Thread(new MyTask()).start();

// Lambda 简写
new Thread(() -> System.out.println("任务运行")).start();
```

优势：实现接口不影响继承其他类，任务逻辑与线程机制解耦。`Thread` 类本身就实现了 `Runnable`。

**方式三：实现 Callable 接口**

```java
class MyTask implements Callable<String> {
    @Override
    public String call() throws Exception {
        return "执行结果";
    }
}

FutureTask<String> futureTask = new FutureTask<>(new MyTask());
new Thread(futureTask).start();

String result = futureTask.get();  // 阻塞等待结果
```

与前两种的区别：`Callable` 的 `call()` 方法有返回值且可以抛出异常。`FutureTask` 同时实现了 `Runnable` 和 `Future`，桥接了 Callable 和 Thread。

`Future.get()` 的注意点：会阻塞当前线程直到结果返回，应避免在主线程长时间等待。Java 1.8+ 推荐用 `CompletableFuture` 替代。

**方式四：线程池（生产环境首选）**

```java
ExecutorService executor = new ThreadPoolExecutor(
    4,                          // 核心线程数
    10,                         // 最大线程数
    60L, TimeUnit.SECONDS,      // 空闲线程存活时间
    new LinkedBlockingQueue<>(100),  // 任务队列
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);

// 提交 Runnable（无返回值）
executor.execute(() -> System.out.println("任务"));

// 提交 Callable（有返回值）
Future<String> future = executor.submit(() -> "结果");
```

**为什么生产环境必须用线程池**：
1. 避免频繁创建销毁线程的开销
2. 控制最大并发数，防止系统过载
3. 提供任务队列缓冲，实现削峰
4. 统一管理线程生命周期

**阿里巴巴 Java 开发手册**明确要求：不要显式创建线程（`new Thread()`），必须通过线程池创建。

### 三、操作系统的五状态模型

```
                        ┌─────────────────────────────────┐
                        │         CPU 调度                 │
                        │                                 │
    创建              ┌──┴──┐    时间片用完     ┌───┐    ┌──┴──┐
  ──────→  新建  ──→  │就绪  │ ←────────────── │运行│    │终止  │
                     └──┬──┘   被高优先级抢占   └─┬─┘    └─────┘
                        │          调度           │
                        │                        │
                        │    I/O 请求            │
                        │    获取锁失败           │
                        │    等待事件             ↓
                        │                    ┌───┴──┐
                        │    I/O 完成         │ 阻塞  │
                        └─────────────────── │       │
                          事件发生            └──────┘
                          获取到锁
```

**各状态的触发条件**：

| 转换 | 触发条件 |
|---|---|
| 新建 → 就绪 | 线程创建完成，加入调度队列 |
| 就绪 → 运行 | 被调度器选中，获得 CPU 时间片 |
| 运行 → 就绪 | 时间片用完、被更高优先级线程抢占、主动 yield() |
| 运行 → 阻塞 | 等待 I/O、等待获取锁、调用 sleep/wait/park |
| 阻塞 → 就绪 | I/O 完成、获取到锁、被 notify/notifyAll 唤醒、sleep 超时 |
| 运行 → 终止 | run() 方法执行完毕、抛出未捕获异常、调用 stop()（已废弃） |

**关键理解：就绪 vs 运行**：
- 就绪：线程"万事俱备只欠 CPU"，在调度队列里排队
- 运行：线程正在 CPU 上执行指令
- 这两个状态之间的切换完全由操作系统调度器控制，线程自身无法决定

**关键理解：阻塞的本质**：
线程进入阻塞状态后，会被移出调度队列，**不占用 CPU 时间**。直到等待的事件发生，才会被重新加入就绪队列。这就是为什么 I/O 密集型应用可以用少量线程处理大量连接——线程在等待 I/O 时不消耗 CPU。

### 四、Java 线程的六状态（`Thread.State` 枚举）

```java
public enum State {
    NEW,            // 新建
    RUNNABLE,       // 可运行（包含就绪和运行中）
    BLOCKED,        // 阻塞（等待 synchronized 锁）
    WAITING,        // 等待（无限期）
    TIMED_WAITING,  // 超时等待
    TERMINATED      // 终止
}
```

**状态转换全景图**：

```
                        start()
  NEW ─────────────────────────→ RUNNABLE
                                    │  ↕
                                    │  （操作系统调度：就绪 ⇄ 运行中，
                                    │   JVM 不区分，统一为 RUNNABLE）
                                    │
            ┌───────────────────────┼───────────────────────┐
            │                       │                       │
            ↓                       ↓                       ↓
        BLOCKED               WAITING              TIMED_WAITING
    （等待synchronized锁）  （无限期等待）          （限时等待）
            │                       │                       │
            │ 触发方式：             │ 触发方式：              │ 触发方式：
            │ 进入synchronized      │ Object.wait()         │ Thread.sleep(ms)
            │ 代码块/方法           │ Thread.join()          │ Object.wait(ms)
            │ 但锁被其他线程持有     │ LockSupport.park()    │ Thread.join(ms)
            │                       │                       │ LockSupport.parkNanos()
            │ 恢复方式：             │ 恢复方式：              │ LockSupport.parkUntil()
            │ 获取到锁              │ Object.notify()       │
            │                       │ Object.notifyAll()    │ 恢复方式：
            │                       │ LockSupport.unpark()  │ 超时自动唤醒
            │                       │ join的线程执行完毕      │
            └───────→ RUNNABLE ←────┘                       │
                           ↑                                │
                           └────────────────────────────────┘

                                                  run()结束/异常
                        RUNNABLE ──────────────────→ TERMINATED
```

**三种"等待"状态的区别**（面试必问）：

| 状态 | 触发方式 | 恢复方式 | 是否释放锁 |
|---|---|---|---|
| **BLOCKED** | 等待 synchronized 锁 | 获取到锁 | — （还没拿到锁） |
| **WAITING** | wait() / join() / park() | notify/notifyAll / join 目标完成 / unpark | wait() 释放锁 |
| **TIMED_WAITING** | sleep(ms) / wait(ms) / parkNanos | 超时自动唤醒 | sleep() 不释放锁 |

**常见面试陷阱**：

1. **`sleep()` 和 `wait()` 的区别**：

| 维度 | sleep() | wait() |
|---|---|---|
| 所属类 | Thread | Object |
| 释放锁 | **不释放** | **释放** |
| 使用位置 | 任何地方 | 必须在 synchronized 块内 |
| 唤醒方式 | 超时自动唤醒 | notify/notifyAll 或超时 |
| 用途 | 暂停当前线程 | 线程间通信 |

2. **`start()` 和 `run()` 的区别**：
   - `start()`：创建新线程，新线程执行 `run()` 方法
   - `run()`：在当前线程同步执行，不创建新线程
   - `start()` 只能调用一次，重复调用会抛 `IllegalThreadStateException`

3. **BLOCKED vs WAITING**：
   - BLOCKED：线程**想进去但进不去**（等锁）
   - WAITING：线程**主动选择等待**（主动调用 wait/join/park）

### 五、线程的终止

**为什么 `Thread.stop()` 被废弃**：
`stop()` 会强制终止线程并立即释放所有锁，导致被锁保护的数据可能处于不一致状态。其他线程看到的就是被破坏的数据结构。

**正确的终止方式**：

```java
// 方式一：中断标志
Thread t = new Thread(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        // 执行任务
    }
});
t.interrupt();  // 设置中断标志

// 方式二：volatile 标志位
class MyTask implements Runnable {
    private volatile boolean running = true;
    public void stop() { running = false; }
    public void run() {
        while (running) { /* 执行任务 */ }
    }
}
```

**中断机制**（`interrupt()`）是 Java 推荐的线程终止方式。它只是设置一个标志位，不会强制停止线程。线程在合适的时机检查标志位，自行决定何时退出。

如果线程处于 `WAITING` / `TIMED_WAITING` 状态时被中断，会抛出 `InterruptedException` 并清除中断标志，需要在 catch 块中处理。

## 常见追问与应对

**Q1: 线程池的核心参数有哪些？各自的作用？**
> 七个核心参数：`corePoolSize`（核心线程数，常驻不回收）、`maximumPoolSize`（最大线程数）、`keepAliveTime`（非核心线程空闲存活时间）、`unit`（时间单位）、`workQueue`（任务队列）、`threadFactory`（线程工厂）、`handler`（拒绝策略）。任务提交后先由核心线程处理，核心线程满了进队列，队列满了创建非核心线程，到最大线程数后执行拒绝策略。

**Q2: 为什么 Java 把就绪和运行合并为 RUNNABLE？**
> 因为 Java 运行在 JVM 上，JVM 把底层操作系统的差异抽象掉了。对 Java 程序来说，线程是"可以被调度执行的"，至于当前有没有在 CPU 上跑，JVM 认为这是操作系统的实现细节，不应该暴露给上层。所以用 RUNNABLE 一个状态统一表示"正在运行或等待 CPU 调度"。

**Q3: 线程的上下文切换具体保存了什么？**
> 保存的内容包括：1）通用寄存器（rax, rbx, rcx...）；2）程序计数器（PC，记录执行到哪条指令）；3）栈指针（SP，指向当前线程的栈顶）；4）条件码寄存器（标志位）。这些信息保存在线程的 TCB 中。注意：**不需要保存整个栈**——栈在线程创建时就已经分配好了，切换时只需要切换栈指针即可。

**Q4: 用户级线程和内核级线程的区别？Java 线程属于哪种？**
> 用户级线程由用户态库管理，操作系统无感知，切换快但不能利用多核；内核级线程由操作系统管理，可以利用多核但切换需要系统调用。Java 传统线程（1:1 模型）是内核级线程，每个 Java 线程对应一个操作系统线程。Java 21 引入的虚拟线程（Virtual Thread）是用户级线程，多个虚拟线程映射到少量操作系统线程上（M:N 模型），类似 Go 的 goroutine。
