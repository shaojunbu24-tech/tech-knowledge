# Redis I/O 多路复用与网络 I/O 模型

> 分类：数据库
> 标签：Redis, I/O 多路复用, epoll, select, poll, 同步异步, 阻塞非阻塞, 网络模型, 单线程

## 面试精答

> 适合面试时口述的简洁版本（2-3分钟能说完的要点）

Redis 之所以能用**单线程**处理上万并发连接，核心就是 **I/O 多路复用**。

先搞清几个基本概念：
- **同步 vs 异步**：调用方是否自己等待结果。同步是自己等，异步是别人完成后通知你
- **阻塞 vs 非阻塞**：等待结果时线程是否被挂起。阻塞是挂起（啥也干不了），非阻塞是立刻返回（可以干别的）

**网络 I/O 的演进**（五步理解）：

1. **阻塞 I/O（BIO）**：一个连接一个线程，`read()` 没数据时线程卡住。10000 个连接就要 10000 个线程，资源爆炸
2. **非阻塞 I/O**：`read()` 没数据时立刻返回错误码，但需要**轮询**（不停问"数据来了没"），CPU 空转
3. **I/O 多路复用**：用一个线程同时监听**多个连接**的文件描述符，哪个有数据就处理哪个。这是 Redis 的方案
4. **信号驱动 I/O**：内核数据就绪时发信号通知，应用捕获信号后去读。编程复杂，实战少
5. **异步 I/O（AIO）**：应用发起 `read()` 后立刻返回，内核把数据**拷贝到用户空间后**再通知。真正的不阻塞。Windows 的 IOCP 是成熟实现，Linux 的 AIO 不够完善

**Redis 用的是第三步——I/O 多路复用**。核心思路：一个线程通过 `epoll` 机制同时监听所有客户端连接，哪个连接有命令发过来了就处理哪个，没有的连接根本不花时间。这就是为什么单线程就能处理几万 QPS。

**一句话总结**：Redis 单线程不慢，是因为它从来不在"等"上浪费时间——所有连接统一由内核通知"谁有数据了"，线程只做真正有活干的连接。

## 原理深入详解

> 深入底层原理，理解"为什么"

### 一、先理解一次网络读取经历了什么

当程序调用 `read()` 从 socket 读取数据时，实际经历了**两个阶段**：

```
阶段一：等待数据就绪（数据从网卡 → 内核缓冲区）
阶段二：拷贝数据（从内核缓冲区 → 用户空间缓冲区）

用户进程调用 read()
  │
  ├──── 阶段一：等待数据 ────────────────────┐
  │     内核等网卡把数据收到内核缓冲区         │ 进程状态？
  │                                          │
  ├──── 阶段二：拷贝数据 ────────────────────┤ 取决于 I/O 模型
  │     内核把数据从内核空间拷贝到用户空间      │
  │                                          │
  ↓                                          │
read() 返回，用户进程拿到数据                  ↓
```

**所有 I/O 模型的区别，就在于这两个阶段"进程在干什么"**。

### 二、五种 I/O 模型逐一拆解

#### 1. 阻塞 I/O（Blocking I/O）

**默认模式**，最直觉的方式。

```
用户进程                  内核
  │                        │
  │─── read(socket) ──────→│
  │                        │ ← 阶段一：等数据（进程阻塞在这里）
  │   (进程挂起，           │
  │    啥也干不了)          │
  │                        │ ← 数据到达内核缓冲区
  │                        │ ← 阶段二：拷贝到用户空间
  │←── 返回数据 ───────────│
  │                        │
```

**问题**：一个线程一次只能处理一个连接。`read()` 等数据时整个线程卡死，其他连接只能排队。

**解决方案**：每个连接开一个线程。10000 个连接 → 10000 个线程 → 内存爆炸（每个线程栈 1MB 就是 10GB）+ 上下文切换开销巨大。

#### 2. 非阻塞 I/O（Non-blocking I/O）

把 socket 设为非阻塞模式，`read()` 没数据时立刻返回错误码 `EWOULDBLOCK`。

```
用户进程                  内核
  │                        │
  │─── read() ────────────→│
  │←── EWOULDBLOCK ────────│  ← 数据还没来，立刻返回
  │                        │
  │─── read() ────────────→│
  │←── EWOULDBLOCK ────────│  ← 还是没来
  │                        │
  │─── read() ────────────→│  ← 轮询（while 死循环）
  │←── EWOULDBLOCK ────────│
  │─── read() ────────────→│
  │←── 数据！ ─────────────│  ← 终于来了
```

**问题**：虽然线程不阻塞了，但需要不停轮询，CPU 一直在空转（100% 占用），而且大部分 `read()` 调用都是无用的。比阻塞 I/O 更浪费资源。

#### 3. I/O 多路复用（I/O Multiplexing）—— Redis 的选择

**核心思想**：不再由应用去轮询每个连接，而是**把"哪些连接有数据了"这件事交给内核来告诉你**。

```
用户进程                  内核
  │                        │
  │─── epoll_wait() ──────→│  ← 把所有 socket 交给内核监听
  │                        │
  │   (进程阻塞，            │  内核在监听所有 socket
  │    但只阻塞一个线程)      │  哪个有数据了就记录下来
  │                        │
  │←── [socket3, socket7] ─│  ← 内核返回"有数据的 socket 列表"
  │     有数据了！           │
  │                        │
  │── read(socket3) ──────→│  ← 只读有数据的 socket
  │←── 数据 ───────────────│
  │── read(socket7) ──────→│
  │←── 数据 ───────────────│
  │                        │
  │─── epoll_wait() ──────→│  ← 继续等待下一批
```

**关键优势**：一个线程同时管理几万个连接，只有真正有数据的连接才调用 `read()`，不需要轮询，不浪费 CPU。

#### 4. 信号驱动 I/O（Signal-driven I/O）

进程先注册一个信号处理函数，然后内核在数据就绪时发送 `SIGIO` 信号通知进程。

```
用户进程                  内核
  │                        │
  │── sigaction(注册信号)──→│
  │   (进程继续干别的事)     │
  │                        │ ← 数据到达
  │←── SIGIO 信号 ─────────│  ← 内核发信号
  │── read() ─────────────→│
  │←── 数据 ───────────────│
```

信号驱动 I/O 在 TCP 协议中触发过于频繁（连接建立、数据到达、连接关闭等都会触发），实用价值有限。

#### 5. 异步 I/O（Asynchronous I/O）

**真正的完全不阻塞**——连数据拷贝都由内核完成。

```
用户进程                  内核
  │                        │
  │── aio_read() ─────────→│  ← 发起读取，立刻返回
  │   (进程继续干别的)       │
  │                        │ ← 阶段一：等数据
  │                        │ ← 阶段二：内核自动拷贝到用户空间
  │←── 信号/回调 ──────────│  ← 全部完成后通知
  │   数据已经在缓冲区里了    │
```

**I/O 多路复用 vs 异步 I/O 的关键区别**：
- 多路复用：内核告诉你"数据准备好了"，**你自己去读**（拷贝还是要阻塞）
- 异步 I/O：内核帮你读完并拷贝好，通知你"数据已经放好了"

Linux 的异步 I/O（`io_submit`）在网络 socket 上支持不完善。Windows 的 IOCP 是成熟的异步 I/O 实现。

### 三、I/O 多路复用的三种实现

Linux 提供了三种 I/O 多路复用的系统调用：

#### select

```c
int select(int nfds,                      // 监听的 fd 数量
           fd_set *readfds,               // 可读 fd 集合
           fd_set *writefds,              // 可写 fd 集合
           fd_set *exceptfds,             // 异常 fd 集合
           struct timeval *timeout);      // 超时时间
```

**工作原理**：
1. 把所有要监听的 fd（文件描述符）打包成一个位图（bitmap）
2. 传给内核，内核遍历检查哪些 fd 有事件
3. 返回有事件的 fd 数量，修改位图标记哪些 fd 有事件
4. 用户进程遍历整个 fd 集合，找出哪些有事件

**致命缺陷**：
- **fd 数量上限**：`FD_SETSIZE` 默认 1024，改大需要重新编译内核
- **每次调用要传全量 fd**：从用户态到内核态拷贝整个 fd 集合，O(n) 开销
- **内核线性扫描**：遍历所有 fd 检查事件，O(n) 复杂度
- **返回后要遍历找事件**：不知道哪些 fd 有事件，只能遍历全部，O(n)

```
监听 10000 个 fd，但只有 2 个有事件：
- 传 10000 个 fd 到内核     ← 拷贝开销
- 内核扫描 10000 个 fd      ← CPU 开销
- 返回后遍历 10000 个 fd    ← 又一次遍历
实际有效工作只处理了 2 个 fd，其余 9998 次都是浪费
```

#### poll

```c
int poll(struct pollfd *fds,   // pollfd 结构体数组
         nfds_t nfds,           // 数组长度
         int timeout);          // 超时时间

struct pollfd {
    int fd;           // 文件描述符
    short events;     // 关注的事件
    short revents;    // 返回的事件
};
```

**相比 select 的改进**：
- 用结构体数组替代位图，**没有 1024 的 fd 数量上限**
- 用 `revents` 字段分离输入输出，不需要每次重置

**但核心问题没变**：
- 还是每次传全量 fd 到内核
- 内核还是线性扫描所有 fd
- 返回后还是要遍历全部找事件

**本质**：poll 是 select 的"改版"，解决了 fd 数量限制，但没有解决性能问题。

#### epoll（Linux 的终极方案）

```c
// 创建 epoll 实例
int epfd = epoll_create(1);

// 注册/修改/删除要监听的 fd
epoll_ctl(epfd, EPOLL_CTL_ADD, socket_fd, &ev);

// 等待事件（只返回有事件的 fd）
int n = epoll_wait(epfd, events, max_events, timeout);
```

**epoll 的三大核心改进**：

**1. 红黑树管理 fd**：
- 用红黑树存储所有要监听的 fd，增删查改都是 O(log n)
- 不需要每次传全量 fd——fd 注册一次就行，后续只做增量操作

**2. 就绪队列（双向链表）**：
- 内核维护一个"就绪队列"，只包含有事件的 fd
- `epoll_wait()` 返回时只拷贝就绪队列里的 fd，O(1) 获取结果
- 监听 10000 个 fd，只有 2 个有事件？直接返回这 2 个，不扫描其余 9998 个

**3. 回调机制**：
- 每个 fd 注册时在内核中设置了回调函数
- 网卡收到数据产生硬件中断 → 内核回调通知 → fd 加入就绪队列
- **不需要主动扫描**，被动通知，时间复杂度 O(1)

```
epoll 的数据结构：

红黑树（存所有监听的 fd）     就绪队列（只有有事件的 fd）
    ┌───┐                    ┌───┐
    │fd3│                    │fd3│ ← 有数据
    └─┬─┘                    └───┘
   ┌──┴──┐                   ┌───┐
   │fd1  │fd7                │fd7│ ← 有数据
   └─────┘                   └───┘
   ┌──┬──┐
   │fd5│fd9│fd12│... (共 10000 个)
   └──┴──┘

epoll_wait() → 直接返回 [fd3, fd7]，不扫描红黑树
```

**三种实现的对比**：

| 维度 | select | poll | epoll |
|---|---|---|---|
| fd 上限 | 1024（默认） | 无限制 | 无限制 |
| 内核检查方式 | 线性扫描所有 fd | 线性扫描所有 fd | 回调通知，只返回有事件的 fd |
| 用户找事件 | 遍历全部 fd | 遍历全部 fd | 直接拿到就绪列表 |
| 每次调用 | 传全量 fd | 传全量 fd | 只传一次（epoll_ctl 增量注册） |
| 时间复杂度 | O(n) | O(n) | O(1)（活跃连接数） |
| 适用连接数 | 几十~几百 | 几百~几千 | 几千~几百万 |

### 四、Redis 的 I/O 多路复用实现

Redis 并没有直接用 `epoll`，而是封装了一个统一的抽象层——**ae（A Event）库**：

```
Redis 源码结构：

ae.h / ae.c           ← 事件驱动框架（通用接口）
├── ae_epoll.c         ← Linux: epoll 封装
├── ae_kqueue.c        ← macOS/BSD: kqueue 封装
├── ae_evport.c        ← Solaris: event ports 封装
└── ae_select.c        ← 兜底: select 封装
```

编译时根据操作系统自动选择最优实现：Linux 用 epoll，macOS 用 kqueue，其他用 select。

**Redis 的事件驱动循环**：

```c
// Redis 的核心事件循环（简化）
void aeMain(aeEventLoop *eventLoop) {
    while (!eventLoop->stop) {
        // 1. 等待事件（epoll_wait / kevent / select）
        numevents = aeApiPoll(eventLoop, tvp);

        // 2. 处理文件事件（网络 I/O）
        for (j = 0; j < numevents; j++) {
            if (fe->mask & AE_READABLE)    → 读事件处理器
            if (fe->mask & AE_WRITABLE)    → 写事件处理器
        }

        // 3. 处理时间事件（定时任务）
        processTimeEvents(eventLoop);
    }
}
```

**两种事件**：
- **文件事件（File Event）**：socket 的读写事件，包括客户端连接、接收命令、发送响应
- **时间事件（Time Event）**：定时任务，如 `serverCron()`（每 100ms 执行一次，负责过期 key 清理、Rehash、统计信息等）

**完整处理流程**：

```
1. 客户端连接 Redis
   → epoll 发现 listening socket 有读事件
   → acceptHandler 接收连接，注册该 socket 的读事件

2. 客户端发送命令 SET key value
   → epoll 发现该 socket 有读事件
   → readQueryFromClient 读取命令
   → processInputBuffer 解析命令
   → setCommand 执行 SET 命令
   → 注册该 socket 的写事件

3. 响应就绪
   → epoll 发现该 socket 可写
   → sendReplyToClient 发送 +OK\r\n
   → 取消写事件注册
```

### 五、Redis 6.0 为什么引入多线程 I/O

Redis 的命令执行是单线程的，但 **网络 I/O 在 6.0 变成了多线程**。

**为什么需要多线程 I/O**：
- 单线程瓶颈不在命令执行，而在**网络 I/O**（读写数据、协议解析）
- 网络越来越快（10Gbps、25Gbps），单线程读写网络数据跟不上
- Redis 5.x 的单线程 I/O 在大 value 或大量小命令时成为瓶颈

**多线程 I/O 的设计**：

```
Redis 6.0+ 的线程模型：

主线程（命令执行，仍然单线程）
  │
  ├── I/O 线程 1：读取客户端请求 / 写入响应
  ├── I/O 线程 2：读取客户端请求 / 写入响应
  ├── I/O 线程 3：读取客户端请求 / 写入响应
  └── I/O 线程 4：读取客户端请求 / 写入响应

流程：
1. 主线程 epoll_wait() 拿到就绪 socket
2. 把 socket 分配给 I/O 线程组（并行读取+解析）
3. 等所有 I/O 线程完成读取
4. 主线程串行执行命令（保证原子性）
5. 把响应分配给 I/O 线程组（并行写入）
6. 等所有 I/O 线程完成写入
```

**关键设计决策**：命令执行仍然是单线程，多线程只负责网络读写。这样既提升了 I/O 性能，又不需要加锁（命令执行没有并发），保持了 Redis 的简单性和原子性保证。

**开启方式**：
```conf
io-threads 4               # I/O 线程数（建议 <= CPU 核心数）
io-threads-do-reads yes    # 读操作也用多线程
```

### 六、Redis 为什么单线程还能这么快

面试常问，综合回答：

1. **I/O 多路复用**：一个线程处理几万连接，不在"等"上浪费时间
2. **内存操作**：所有数据在内存中，读写都是纳秒级
3. **高效的数据结构**：SDS（简单动态字符串）、跳表、压缩列表、整数集合等
4. **避免上下文切换**：单线程没有锁、没有线程切换开销
5. **避免锁竞争**：多线程加锁/解锁的开销在高频操作中可能比单线程还大
6. **单线程心智模型简单**：所有命令原子执行，不需要考虑并发问题

### 七、操作系统层面的知识补充

**文件描述符（File Descriptor, fd）**：
- Linux 中一切皆文件，socket 也是一个文件，有对应的 fd（非负整数）
- 进程通过 fd 来操作 socket
- 每个进程默认最多打开 1024 个 fd（`ulimit -n` 可查看/修改），生产环境通常调大到 65535+

**用户态和内核态**：
- **用户态**：应用程序运行的空间，不能直接访问硬件
- **内核态**：操作系统内核运行的空间，可以访问所有硬件和内存
- `read()` / `select()` / `epoll_wait()` 等系统调用会触发**从用户态切换到内核态**，有上下文切换开销
- 这就是为什么 select 每次传全量 fd 开销大——每次都要从用户态拷贝到内核态

**网卡中断**：
- 网卡收到数据后产生**硬件中断**，通知 CPU
- CPU 暂停当前工作，执行中断处理程序
- 中断处理程序把数据拷贝到内核缓冲区，然后通知等待的进程
- epoll 的回调机制就是建立在这个中断机制之上的

## 常见追问与应对

**Q1: select、poll、epoll 面试怎么快速说清区别？**
> select 有 fd 上限（1024）且每次全量扫描 O(n)；poll 去掉了上限但扫描方式不变；epoll 用红黑树管理 fd、回调通知就绪、只返回有事件的 fd，O(1) 性能，是唯一能支撑百万级连接的方案。

**Q2: 既然 epoll 这么好，为什么 Redis 还要封装 ae 库？**
> 跨平台兼容。epoll 是 Linux 特有的，macOS 用 kqueue，Windows 用 IOCP。ae 库把这些差异封装起来，Redis 上层代码不需要关心底层用的是什么系统调用。编译时自动选择最优实现。

**Q3: Redis 6.0 多线程和 Memcached 多线程有什么区别？**
> Memcached 是真正的多线程——多个线程独立处理请求，需要加锁保护共享数据。Redis 6.0 只是 I/O 多线程——网络读写并行，命令执行仍然单线程串行，不需要加锁。Redis 的方案更简单，也能保证命令的原子性。

**Q4: Redis 4.0 的后台线程是干什么的？和 6.0 的多线程有什么区别？**
> Redis 4.0 引入了后台异步线程（bio），专门处理**阻塞型操作**：1）异步删除大 key（`UNLINK`）；2）AOF 的 fsync 刷盘；3）关闭大文件。这些操作如果放在主线程会阻塞事件循环。6.0 的多线程是**网络 I/O 线程**，专门加速网络读写，两者解决的是不同问题。
